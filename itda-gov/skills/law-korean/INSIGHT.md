# law-korean INSIGHT

> 마지막 갱신: 2026-03-30

## 이번 작업에서 확인한 인사이트

### 1. Phase 3의 가장 작은 안전한 시작점은 `oldAndNew` 목록 조회다

- 공식 가이드가 명확하고, 기존 `search_*` 패턴에 바로 맞출 수 있다.
- 상세 조회보다 파싱 난도가 낮고, CLI/테스트 구조를 그대로 재사용 가능하다.

### 2. `oldAndNew` 계열은 실API 응답 기준으로 바로 쓸 수 있다

- 목록: `lawSearch.do?target=oldAndNew`
- 상세: `lawService.do?target=oldAndNew`
- 실제 OC로 호출 시 `근로기준법` 기준 정상 응답 확인

### 3. 실API 검증은 반드시 필요하다

- 문서만 보면 알기 어려운 응답 특성이 실제 호출에서 드러났다.
- 대표 사례: `소관부처명`이 동일 값 반복으로 내려와 출력 품질이 깨졌다.
- 이 문제는 재현 테스트를 먼저 추가한 뒤 정규화 로직으로 해결했다.

### 4. law-korean의 확장 패턴은 이미 충분히 안정적이다

- 새 기능은 `*_api.py` + `search_*.py`/`get_*.py` + `tests/test_*.py` 조합으로 무리 없이 붙는다.
- 공통 재사용 포인트:
  - `_fetch_xml()`
  - `resolve_oc()`
  - `cache_get/cache_set/cache_key()`
  - `print_results_table()` / Markdown formatter

### 5. 다음 우선순위는 상세 조회가 자연스럽다

- 이미 `oldAndNew` 상세 엔드포인트와 XML 구조를 실API로 확인했다.
- 따라서 다음 단계는 `get_old_and_new.py` + 상세 파서 + 테스트가 가장 자연스럽다.

### 6. `oldAndNew` 상세는 `ID`와 `MST`를 둘 다 받아야 한다

- 공식 가이드 기준으로 `ID` 또는 `MST` 중 하나가 필수다.
- 실제 구현도 이 규칙을 그대로 반영하는 편이 검색 결과와 상세 조회를 자연스럽게 이어 준다.

### 7. 상세 응답의 핵심은 "비교 diff"가 아니라 "구법/신법 조문 묶음"이다

- 공식/실응답 기준으로 상세는 `구조문목록/조문`과 `신조문목록/조문`의 병렬 목록이다.
- 따라서 2차 슬라이스는 diff 알고리즘보다 **정확한 수집과 보기 좋은 출력**에 집중하는 것이 맞다.

### 8. CDATA 안의 `<P>` 마크업은 최소 정리만 하는 편이 안전하다

- `<P>`/`</P>` 제거만으로도 가독성은 충분히 개선된다.
- `<신 설>` 같은 표지는 보존해야 하므로, 모든 꺾쇠 태그를 일괄 제거하면 안 된다.

### 9. 조문 비교 MVP는 기존 `get_law_detail()` 재사용이 가장 경제적이다

- 별도 API를 만들지 않고 `resolve_law_id()` + `get_law_detail(article_no=...)` 조합으로 바로 비교 가능하다.
- 그래서 1차 조문 비교는 새 백엔드 모듈 없이 CLI 한 개로 충분했다.

### 10. 조문 비교 1차는 text + unified diff만으로도 실용성이 있다

- `difflib.unified_diff`만으로도 삭제/추가 흐름을 바로 볼 수 있다.
- 처음부터 JSON/Markdown 구조화까지 넣기보다, text MVP로 빠르게 검증한 뒤 확장하는 편이 맞다.

### 11. live smoke test용 법령/조문 조합은 신중히 골라야 한다

- 서로 다른 법령의 임의 조문 번호를 비교하면 장 제목/절 제목처럼 기대와 다른 텍스트가 나올 수 있다.
- 따라서 다음 단계에서는 비교 품질을 보여주기 좋은 대표 조합을 별도로 확보하는 것이 좋다.

### 12. 조문 비교 2차는 출력 확장만으로 충분히 독립적인 슬라이스가 된다

- 비교 로직 자체는 1차 MVP와 동일하게 유지할 수 있다.
- 결과를 `left/right/diff_lines/has_diff` 구조로 한 번 정리하면 text/json/md 분기가 단순해진다.

### 13. Markdown 출력은 fenced diff block이 가장 자연스럽다

- 일반 bullet보다 ```diff 블록이 추가/삭제 흐름을 그대로 보여 주기 쉽다.
- 따라서 2차 슬라이스의 md 출력은 side-by-side보다 diff block 중심이 맞다.

### 14. 조문 비교에서 법령명은 fuzzy보다 exact 매칭이 낫다

- `search_laws()`의 일반 검색 결과는 입력과 다른 법령을 반환할 수 있어 오류 메시지가 흐려질 수 있다.
- 조문 비교처럼 양쪽 컨텍스트가 중요한 도구는 이름 입력 시 **정확히 일치하는 법령명만 채택**하는 편이 안전하다.

### 15. 대표 예시는 같은 법령의 인접 조문이 가장 안정적이다

- `근로기준법 제60조 ↔ 제61조`처럼 같은 법령 내 연관 조문은 live demo 품질이 안정적이다.
- 반대로 서로 다른 법령의 임의 조합은 장/절 제목이나 맥락 차이로 비교 품질이 흔들릴 수 있다.

### 16. lstrmAI는 “법령용어 검색”이고, 실제 연계 키는 MST다

- 실응답에서 법령용어 요소의 `id` 속성보다 `용어간관계링크` 안의 `MST`가 다음 단계 연계에 더 중요했다.
- 그래서 검색 결과에는 `mst`를 함께 노출하는 편이 실무적으로 유용하다.

### 17. lstrmAI 1차 슬라이스는 search-only로 충분하다

- `target=lstrmAI`만으로도 용어명, 동음이의어 여부, 비고, 관계 링크를 확보할 수 있다.
- 따라서 1차는 `search_lstrm.py`까지만 넣고, 관계 상세는 `lstrmRlt` 슬라이스로 분리하는 것이 가장 안전하다.

### 18. lstrmRlt는 MST 기반 상세 조회로 묶는 편이 자연스럽다

- `lstrmAI` 응답의 링크 안에 이미 `MST`가 포함되어 있어, 다음 단계 CLI는 `--mst` 하나만 받는 형태가 가장 단순했다.
- 검색어 기반 재조회보다 `search_lstrm → get_lstrm_rlt` 흐름이 더 명확하고 재현 가능했다.

### 19. lstrmRlt 실응답은 “법령용어 ↔ 일상용어 관계 목록”이다

- 실응답 기준으로 핵심은 `일상용어명`, `용어관계코드`, `용어관계`, 링크들이다.
- 따라서 1차 출력은 관계 표 중심이면 충분하고, 조문 연계(`lstrmRltJo`)는 다음 슬라이스로 분리하는 것이 맞다.

### 20. lstrmRltJo는 조문 본문까지 주기 때문에 바로 실무 활용이 가능하다

- 실응답에는 `법령명`, `조번호`, `조문내용`, `용어구분`, `조문연계용어링크`가 함께 포함된다.
- 따라서 별도 `get_law.py` 호출 없이도 “이 용어가 실제 어느 조문에서 어떻게 쓰였는가”를 바로 보여줄 수 있다.

### 21. 조문연계용어링크에서 `ID`와 `JO`를 뽑아두는 편이 다음 단계 확장에 유리하다

- `조문연계용어링크` 안에는 `ID`와 `JO`가 들어 있어 역참조나 후속 법령 조회에 바로 활용할 수 있다.
- 그래서 1차 파서에서 `law_id`, `jo`를 같이 노출해두는 것이 이후 `joRltLstrm` 확장에 유리하다.

### 22. joRltLstrm은 `ID + JO`를 그대로 받는 얇은 래퍼가 가장 자연스럽다

- `lstrmRltJo` 실응답이 이미 `법령ID/JO`를 제공하므로, 역방향 조회는 그 값을 그대로 받는 편이 가장 단순하다.
- 검색어 재해석이나 추가 변환보다 링크 재사용이 더 안정적이었다.

### 23. 법령용어 연계는 이제 양방향 그래프처럼 다룰 수 있다

- `lstrmAI` → `lstrmRlt` / `lstrmRltJo`
- `lstrmRltJo` → `joRltLstrm`
- 이 구조가 갖춰지면서 법령용어, 일상용어, 조문 사이를 자연스럽게 왕복 탐색할 수 있게 됐다.

### 24. joRltLstrm은 검색형이 아니라 링크 재활용형 상세 조회다

- 공식 가이드 기준으로 `ID`와 `JO`가 핵심 파라미터이고, 실제로 `lstrmRltJo` 응답이 그 값을 제공한다.
- 따라서 별도 탐색 UX보다 `get_lstrm_rlt_jo → get_jo_rlt_lstrm` 흐름이 가장 자연스럽다.

### 25. 조문-용어 역방향 조회도 출력 길이 이슈가 바로 생긴다

- 실API 기준으로 한 조문에 연결된 용어가 20개 이상 붙는 사례가 있었다.
- 다음 단계에서 summary/truncate를 넣을 필요성이 더 분명해졌다.

### 26. 길이 제어는 hard cut보다 `--summary-only`가 먼저다

- `lstrmRltJo`, `joRltLstrm`은 본문이 길어지는 주된 원인이 조문 텍스트이므로, 우선 `--summary-only`로 본문만 숨기는 편이 가장 단순했다.
- 관계/용어 목록은 그대로 유지하되, 조문 본문만 제거해도 출력량이 크게 줄었다.

### 27. 트런케이션은 JSON에도 동일하게 적용하는 편이 안전하다

- 긴 조문 내용을 JSON으로 그대로 내보내면 후속 에이전트 입력이 과도해질 수 있다.
- 그래서 30,000자 제한은 text/md뿐 아니라 json 경로에도 동일하게 적용하는 것이 맞다.

### 28. 긴 관계 목록은 `--max-items` 기본값으로 먼저 줄이는 편이 실용적이다

- `lstrmRltJo`, `joRltLstrm`는 본문을 숨겨도 항목 수 자체가 많아 여전히 길 수 있다.
- 그래서 기본 20개 제한을 두고, 필요하면 `--max-items`를 늘리는 방식이 가장 단순했다.

### 29. 목록 제한 안내는 포맷별로 다르게 주는 편이 기존 패턴과 맞다

- text는 본문 하단 안내문, md는 blockquote 안내, json은 stderr 안내가 `get_law.py` 패턴과 가장 유사했다.
- 그래서 이번 슬라이스도 같은 방식으로 맞췄다.

### 30. lstrmRltJo / joRltLstrm 중복은 실제 API 응답 단계에서 정리해야 한다

- 같은 조문이나 같은 법령용어가 실응답에서 반복되는 사례가 확인됐다.
- CLI 출력 단계가 아니라 `lstrm_api.py` 파서 단계에서 제거해야 캐시·json·md·text 전 경로가 한 번에 정리된다.

### 31. 중복 키는 표시용 이름보다 구조적 키가 안전하다

- `lstrmRltJo`는 `law_id + jo + term_type_code`, `joRltLstrm`는 `mst + term_type_code` 조합으로 중복 제거가 안정적이었다.
- 단순히 이름만 비교하면 서로 다른 항목이 잘못 합쳐질 위험이 있다.

### 32. 중복 제거는 API 레이어에서 하는 편이 맞다

- `lstrmRltJo`, `joRltLstrm`의 중복은 실응답 자체에서 발생하므로, CLI가 아니라 API 파서에서 제거하는 편이 캐시/JSON/MD/text 전 경로에 동시에 적용된다.
- 그래서 dedupe는 `lstrm_api.py`에서 처리하고, CLI는 그대로 두는 쪽이 더 단순했다.

### 33. dedupe 키는 링크 기반 식별자가 가장 안정적이다

- `lstrmRltJo`는 사실상 `law_id + jo + term_type_code`,
- `joRltLstrm`는 `mst + term_type_code` 조합이 중복 제거에 충분했다.
- 단순 이름만으로 dedupe하면 서로 다른 항목이 잘못 합쳐질 위험이 있다.

### 34. lstrmRlt도 같은 구조적 키 기반 dedupe가 맞다

- `lstrmRlt`는 `related_mst + relation_code` 조합으로 중복 제거하는 것이 가장 단순했다.
- 이렇게 해서 법령용어 관계 계열 전체가 같은 dedupe 원칙을 공유하게 됐다.

### 32. 조문 비교는 제목과 diff 요약만 추가해도 가독성이 크게 좋아진다

- 본문 전체 diff를 그대로 두더라도, 조문 제목이 있으면 맥락 파악이 빨라진다.
- 추가/삭제 줄 수를 먼저 보여 주면 사용자가 diff 크기를 바로 가늠할 수 있어 출력 노이즈가 줄어든다.

### 33. compare_articles도 `--summary-only`가 있으면 긴 diff를 빠르게 판단할 수 있다

- 조문 비교는 본문과 raw diff가 모두 길어질 수 있어, 요약만 먼저 보는 모드가 유용하다.
- 제목 + diff 요약만 남기면 "이 비교를 더 깊게 볼지"를 빠르게 결정할 수 있다.

### 34. raw diff는 줄 수 제한만 있어도 체감 노이즈가 크게 줄어든다

- `difflib.unified_diff` 자체는 유지하되, `--max-diff-lines`만 둬도 과도한 출력 문제를 크게 줄일 수 있었다.
- 그래서 diff 알고리즘을 바꾸기 전에 출력량 제어부터 넣는 것이 가장 안전한 다음 단계였다.

## 작업 원칙

- 먼저 구현하지 말고, **실API 관찰 → 재현 테스트 → 최소 구현** 순서로 간다.
- 공식 문서보다 실응답을 우선 기준으로 삼되, 문서와의 차이는 테스트로 고정한다.
- 문서/README 갱신은 기능 슬라이스가 안정화된 뒤 반영한다.
