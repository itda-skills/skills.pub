# problem-guide — 막연한 문제를 구체적 지시로 바꿔주는 가이드 생성 스킬

## 핵심 철학 — 막연한 문제 → 막연한 결과

대부분의 "원하는 대로 안 나온다"는 **도구의 한계가 아니라 지시의 구체성 부족**에서 온다. 사람은 머릿속 그림을 다 말하지 않은 채 결과만 기대하고, AI는 빈칸을 임의로 채운다. 이 스킬의 역할은 두 가지다.

1. **역질문으로 문제를 좁힌다** — "무엇을, 어떤 데이터로, 어떤 결과 형태로, 어떤 기준으로"를 끌어낸다.
2. **두 환경의 구체적 단계 가이드를 만든다** — 같은 문제를 Cowork(제약)와 Claude Code(자유)에서 각각 어떻게 푸는지, 그리고 언제 외부 솔루션을 조합하는지 보여준다.

목표는 답을 대신 주는 게 아니라, 사용자가 **"이렇게 구체적으로 지시하면 되는구나"를 체득**하게 하는 것이다.

## 입력

막연한 문제/불편 한 줄. (예: 사전설문 '아쉬운 점', 강의 중 "이거 안 돼요" 호소)

## 모드 — 대면 vs 부재(배치)

- **대면 모드**: 사용자가 함께 있으면 1단계 역질문을 실제로 주고받아 구체화한 뒤 가이드를 만든다.
- **부재 모드(배치)**: 사용자가 자리에 없는데 문제만 받은 경우(주로 **강의 운영자**가 사전설문 페인을 사람별 가이드로 일괄 생성할 때 — 일반 사용자 경로가 아니다). 이때는 역질문을 던질 수 없으므로 **(a) 확인 질문 목록 + (b) 가정을 명시한 잠정 재정의 + (c) 가장 그럴듯한 해석 기준의 2트랙 잠정 가이드**를 만들고, 문서 말미에 "세션에서 확인 후 보정"을 명시한다. 모든 가정은 *가정*으로 표시한다.

## 워크플로우

### 1단계 — 역질문으로 구체화 (가장 중요, 건너뛰지 말 것)

문제를 좁히는 질문 **3~6개**를 한 번에 제시한다. 아래 5축에서 빠진 것을 골라 묻는다.

- **목표 결과물**: 정확히 어떤 산출물? (형식·길이·용도·볼 사람)
- **입력 데이터**: 무엇을 갖고 있나? (형식·규모·민감도)
- **현재 시도**: 지금 어떻게 했고 어디서 어긋났나? (실제 프롬프트·결과)
- **품질 기준**: "원하는 대로"의 구체 기준은? (좋은 예시 1개를 끌어낸다)
- **제약**: 보안(사내 데이터)·환경·반복 여부

답이 오면 모호한 부분만 1라운드 더 좁힌다. **사용자가 답하기 어려워하면, 선택지(A/B/C)로 제시**해 고르게 한다.

### 2단계 — 문제 재정의

답변을 모아 **"진짜 문제"를 1~2문장으로 재진술**하고, 왜 막연했는지(보통 어떤 축의 구체성이 빠졌는지)를 한 줄로 진단한다.

### 3단계 — 두 갈래 단계별 가이드

같은 문제를 두 환경에서 푼다. **각 트랙에 "막연한 지시 → 구체적 지시"의 대비 예시를 반드시 포함**한다. 또한 **사용자 수준에 맞춰 주(主)·보(補)를 정한다** — 비개발자/입문자는 Track A(Cowork)를 주력으로 두고 Track B는 "더 키우고 싶을 때"로, 코드·자동화 경험이 있는 상급자는 Track B(Code)를 주력으로 깊게 다룬다.

> **비개발자는 Track A(Cowork)만 보면 됩니다.** Track B(Code)는 개발 경험자가 자동화를 더 키울 때 쓰는 거라 **지금은 몰라도 괜찮습니다** — 비개발자 산출물에서는 Track B를 한 줄로만 언급하고 접어 둔다.

**Track A · Claude Cowork (제약 환경)**
- 특성: 채팅 + 워크스페이스 호스트 폴더. 코드 실행은 되지만 영속 환경·외부 시스템 연동·반복 자동화는 제한적. 비개발자 친화.
- 가이드: ① 워크스페이스 폴더에 둘 자료(샘플 데이터·원하는 결과 스케치·양식), ② 순서대로 칠 **구체적 프롬프트**(인터랙션·형식·라이브러리까지 명시), ③ 한 번에 다 요구하지 말고 단계적 보정 루프.

**Track B · Claude Code (자유 환경)**
- 특성: 파일시스템·코드 실행·반복·버전관리·외부 도구/MCP 연동 자유. 더 크고 반복적이고 자동화된 해법.
- 가이드: ① 실제 프로젝트 스캐폴드, ② 실데이터 바인딩·스크립트화·자동 반복, ③ 재사용 템플릿·정기 실행.

### 4단계 — 솔루션 조합 제안 (상황에 따라)

문제의 **데이터 규모·반복성·연동 필요**가 기본 도구를 넘어서면 외부 솔루션 조합을 트리거 기준과 함께 제안한다.

- **대용량·SQL 집계** → 전용 분석 DB/웨어하우스(BigQuery 등) (집계는 거기서, 결과만 Claude로 가져와 가공·시각화). 비개발자는 우선 표본 CSV로 충분한지부터 본다. ⚠️ BigQuery 등은 **사외 클라우드 서비스** — 사내 데이터를 올리려면 회사 보안 정책·승인을 먼저 확인한다
- **정기 반복** → 스크립트화 + 스케줄(Code 환경)
- **대화형 시각화** → 차트 라이브러리(Chart.js/Plotly 등) 또는 BI 도구
- **사내 시스템 데이터** → 비식별·표본 추출 후 처리(보안 우선)

"언제 굳이 이걸 더하나"를 명확히 — 작은 문제에 과한 솔루션을 권하지 않는다.

### 5단계 — 출력: 가이드 문서(.md)

구조: **문제 재정의 / 막연→구체 지시 대비 / Cowork 단계 / Code 단계 / 솔루션 조합(해당 시) / 체크리스트**.

## 흔한 문제 유형 → 진단 빠른 참조

표면 호소는 달라도 원인은 몇 갈래로 묶인다. 진단을 빠르게 하기 위한 매핑.

| 표면 호소 | 흔한 진짜 원인 | 핵심 처방 |
|---|---|---|
| "원하는 답이 안 나와 반복" | 결과물 형식·예시 미지정, 한 번에 큰 요구 | 형식·예시 선합의 + 단계 분할 |
| "일관성이 없다" | 매번 다른 지시, 기준 부재 | 명세/템플릿으로 기준 고정 |
| "오답이라 크로스체크"·"우기는 경향" | 근거 없는 단정(환각) | 출처 강제·실행으로 답하기·자기검증 |
| "대용량 작업 누락" | 한 번에 처리 | 분할 처리 + 건수 대조 |
| "컨텍스트·지식저장 미흡" | 다 대화에 넣음 | 지식 파일화 + 단계 적재 + 검색계층 |
| "산출물(HTML 등)이 원하는 대로 안 됨" | 출력 스펙 미지정 | 결과 스펙·예시 명시 + 단계적 |

## 작동 예시 — 인터랙티브 HTML 보고서 (감사 직군)

> 입력 문제: "인터랙티브 기능이 들어간 HTML 보고서를 생성해도 원하는 대로 작성되지 않을 때가 있음."

**1단계 · 역질문 (이렇게 묻는다)**
1. 어떤 인터랙션을 원하시나요? (예: 부서/기간 필터, 차트 hover, 행 클릭 시 상세 펼침, 정렬)
2. 데이터는 무엇인가요? (형식·대략 규모 — 예: 월별 부서 예실 CSV 2천 행)
3. 원하는 화면을 한 장 스케치하거나, 참고할 대시보드가 있나요?
4. 누가 보는 보고서인가요? (경영진 1page 요약 vs 분석가용 상세)
5. 지금까지 어떤 문장으로 요청하셨고, 어디서 어긋났나요?

**2단계 · 문제 재정의 (가정 답변 기준, 예시)**
> "월별·부서별 예실 데이터(약 2천 행, CSV)를 **부서 필터 + 예실 막대/추이 차트 + 행 클릭 시 상세가 펼쳐지는 1페이지 HTML 대시보드**(사내 공유용)로 만들고 싶다."
> 진단: 막연했던 이유 = '인터랙션 종류·데이터 구조·레이아웃 기준'이 지시에 없어 매번 다르게 나옴.

**3단계 · 막연 → 구체 대비**
- ❌ 막연: "예실 데이터로 인터랙티브 HTML 보고서 만들어줘."
- ✅ 구체: "첨부한 `예실.csv`(컬럼: 부서/월/예산/실적)로 1페이지 HTML 대시보드를 만들어줘. 상단에 부서 드롭다운 필터, 가운데 월별 예산 대 실적 **막대 그래프**(Chart.js), 하단에 표 — 행을 클릭하면 그 부서·월의 차이 원인 메모가 펼쳐지게. 색은 회사 톤(남색/회색), 외부 CDN만 사용. 먼저 정적 버전부터 만들고 보여줘."

**Track A · Cowork 단계**
1. 워크스페이스 폴더에 `예실.csv`(비식별)와 원하는 화면 손그림/설명 메모를 둔다.
2. 위 ✅ 구체 프롬프트로 **정적 버전**부터 요청 → 미리보기 확인.
3. "이제 부서 드롭다운으로 차트·표가 필터되게 추가해줘" → 한 기능씩 추가.
4. "행 클릭 시 상세가 토글되게" → 인터랙션 보정.
5. 스타일·여백만 마지막에 손본다. (한 번에 다 요구하지 않는 게 핵심)

**Track B · Claude Code 단계**
1. `report/` 폴더에 `index.html`+`app.js`+샘플 데이터로 실제 프로젝트 스캐폴드.
2. 실데이터를 바인딩하고 로컬 미리보기로 **반복**하며 인터랙션 완성(라이브 새로고침).
3. 데이터만 바꾸면 매월 재생성되도록 **템플릿화**(같은 보고서를 매월 1초 재생산).
4. git으로 버전관리 — "지난달 버전과 비교".

**4단계 · 솔루션 조합 (이 경우)**
- 데이터가 2천 행이면 조합 불필요(CSV로 충분).
- 만약 **수백만 행·여러 시스템 집계**라면 → BigQuery에서 부서·월 집계 쿼리를 돌리고, **집계 결과(수십 행)만** HTML 대시보드에 피드. 매월 정기면 BQ 쿼리→HTML 생성을 Code에서 스크립트로 자동화.

**5단계 · 체크리스트**
- [ ] 인터랙션 종류를 다 적었는가
- [ ] 데이터 컬럼 구조·샘플을 줬는가
- [ ] "정적 먼저 → 인터랙션 추가" 단계로 갔는가
- [ ] 사내 데이터는 비식별했는가

## 사용 팁

- 이 스킬은 **답을 주는 도구가 아니라 질문을 주는 도구**다. 1단계 역질문을 생략하면 효과의 절반이 사라진다.
- 사용자가 비개발자면 Track A(Cowork)를 기본으로, Track B(Code)는 "더 키우고 싶을 때"로 안내한다.
- 보안: 사내 원장·전표·미공개 자료는 비식별·표본화 후 사용. 공개/가상 데이터 우선. (비식별 = 거래처·이름·실제 금액을 가짜 값으로 바꾸기 — 비개발자용 쉬운 설명·체크리스트는 플러그인 `GUIDE.md` 5절 참조.)
- 산출물은 항상 **초안**이며 사용자의 검증·판단이 최종이다.
