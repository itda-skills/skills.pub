# 예시: 공학 R&D — 알고리즘 구현 + 검증을 한 흐름으로

> miniskill-forge 도출 시 직군 근접 예시로 참조하는 일반화본(참가자 고유정보 제거).

## 적합 상황
- 누가: 공학 알고리즘·데이터를 다루는 R&D 담당자
- 반복 작업: 수식·의사코드를 코드로 옮기고 → 경계조건 확인 → 알려진 정답·독립 재구현으로 교차검증 → 단위 테스트로 굳히는 흐름
- 굳히면 좋아지는 것: 신뢰성(추론이 아니라 실제 실행 결과로 답하게 강제 → 환각·근거 없는 단정 차단) + 일관성(구현과 검증을 한 묶음으로 매번 같은 기준)

## 워크스페이스에 둘 자료 (있는 것만, 공개 가능·합성 자료만)
- 선호하는 코드 스타일·헤더 주석 양식이 담긴 과거 코드 1~2개(출력 코드 기준)
- 자주 쓰는 상수·단위·좌표계 정의표(예: 지구 타원체 파라미터, 단위 환산표) — 출처 URL 포함
- 검증 기준으로 쓸 알려진 정답 케이스·벤치마크 메모(해석해, 표준 예제). 미공개·보안 과제 자료는 두지 않음

## 도출 프롬프트 (Cowork 채팅에 붙여넣기)
> "나는 공학 분야 연구자입니다. 공학용 알고리즘을 구현할 때마다 구현과 검증을 한 흐름으로 처리하는 작업을 반복합니다. 입력은 '수식 또는 의사코드 1개'와 선택적으로 '알려진 정답·벤치마크 케이스'를 받습니다. 다음 순서로 처리해줘: (1) 코드를 구현하고 단위·좌표계 가정을 헤더 주석에 명시, (2) 실제로 실행해 입력·출력 예시 제시(추론으로 답하지 말 것), (3) 경계조건(0·음수·빈 입력·NaN·대값)을 직접 입력해 동작 표로 정리, (4) 알려진 정답이나 독립 재구현으로 교차검증해 오차를 수치로 제시, (5) 단위 테스트(pytest)를 작성·실행해 로그 첨부. 제약: 확실하지 않으면 단정하지 말고 '확인 불가'로 답할 것, 확인된 사실과 추정을 분리 표기할 것, 사용한 상수·표준값은 출처 URL을 표기할 것, 미공개·보안 데이터는 다루지 말 것. 이 흐름을 SKILL.md 1개로 정리해줘 — 트리거·입력 가정·워크플로우·출력 포맷 섹션 포함."

## 도출될 SKILL.md 골격
- name: `algo-implement-verify`
- 트리거: "이 알고리즘 구현하고 검증해줘", "이 수식 코드로 만들고 확인해줘", "경계조건·교차검증까지 해줘"
- 워크플로우 핵심: (1) 코드 구현 + 단위·좌표계 가정 헤더 주석 → (2) 실제 실행해 입출력 예시(추론 금지) → (3) 경계조건(0·음수·빈 입력·NaN·대값) 직접 입력해 동작 표 → (4) 알려진 정답/독립 재구현과 교차검증, 오차 수치 제시 → (5) pytest 작성·실행, 로그 첨부 → (6) 불확실은 '확인 불가', 사실/추정 분리
- 출력 포맷: 구현 코드(헤더 가정) + 실행 결과(입력→출력) + 경계조건 표 + 교차검증(방법·결과·오차) + 단위 테스트 로그(통과/실패) + 확인된 사실 vs 추정. 상수·표준값은 출처 URL, 공개·합성 자료만

## 사용 예시 프롬프트
- "이 의사코드(측지좌표→ECEF 변환)를 구현하고 경계조건·교차검증·단위테스트까지 해줘."
- "이 수식(저역통과 필터 계수 계산)을 코드로 만들고, 알려진 차단주파수 케이스로 검증해줘."
- "이 상태추정 알고리즘을 다른 방법으로도 독립 구현해서 두 결과를 교차검증해줘. 다르면 단정하지 말고 원인 후보를 나열해줘."
