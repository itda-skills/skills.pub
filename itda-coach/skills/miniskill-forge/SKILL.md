# miniskill-forge — 반복 업무를 재사용 미니스킬로 굳히기

## 이 스킬이 푸는 문제

같은 형식의 작업을 할 때마다 **긴 프롬프트를 다시 복사**하고, 그러다 보니 사람마다·주마다 결과 형식과 깊이가 달라진다. "이 작업은 항상 이렇게 해"를 한 번 박아두면, 다음부터 **한 마디**로 같은 품질이 나온다. 이 스킬은 그 "한 번 박기"를 옆에서 이끈다 — 참가자가 직접 자기 Cowork에 미니스킬(SKILL.md)을 도출하도록.

## 미니스킬이란

"이 작업은 항상 이렇게 처리해"를 적어 둔 작은 설명서(SKILL.md). 한 번 만들어 두면 매번 긴 프롬프트 대신 "○○ 돌려줘" 한 마디로 같은 결과를 얻는다.

- **언제 만드나**: 같은 형식의 작업을 주 1회 이상 반복할 때.
- **무엇이 좋아지나**: 형식 일관성(누가 쓰든 같은 모양), 검증 누락 방지(절차가 스킬에 박힘), 인수인계 용이.
- ⚠️ 도출하는 것은 "완성 SKILL.md 복붙"이 아니라, **Cowork에 절차를 설명해 스스로 만들게** 하는 것이다. 본인 양식·용어로 나와야 진짜 본인 스킬이 된다.

## 시작 전 — Cowork 환경 주의 (꼭 먼저)

**쉽게 말해 '워크스페이스'는 내 컴퓨터의 폴더 하나다** — Claude에게 그 폴더를 보여주고, 자료는 거기에 **파일로 둔다**(비개발자용 풀이는 플러그인 `GUIDE.md` 4절).

**Cowork에는 별도 "Project Knowledge" 업로드 UI가 없다.** claude.ai 프로젝트 메뉴·"지식 추가"·업로드 버튼을 안내하지 말 것 — 그건 claude.ai 습관이고 Cowork엔 그 UI가 없다(호스트 폴더를 워크스페이스로 마운트=연결). 실데이터는 사내 보안 정책대로, 제작 단계는 마스킹 표본만 쓴다.

## 진행 절차

### 0단계 — 대상 작업 1개 특정
"직군 + 가장 반복적인 작업"을 한 줄로. 막막하면 hour-slice·problem-guide에서 넘어온 조각을 그대로 쓴다. 한 번에 하나만(여러 개면 가장 자주·가장 지겨운 것부터).

### 1단계 — 직군 근접 예시 보여주기
`references/examples/`의 예시 뱅크에서 사용자 직군에 가까운 **1개**를 골라 보여준다. **어느 게 가까운지 AskUserQuestion**으로 고르게 해도 좋다. 예시는 "이런 모양으로 나온다"는 그림을 잡아줄 뿐, 그대로 복붙하지 않는다.

### 2단계 — 워크스페이스 자료 구성
Cowork 워크스페이스(호스트 폴더)에 둘 자료를 안내한다(있는 것만): 과거 산출물 양식 1부, 마스킹한 표본 1~2건, 규칙/체크 메모.

> (위 '환경 주의'대로 자료는 호스트 폴더에 **파일로** 둔다 — 업로드 UI 없음.)

### 3단계 — 도출 프롬프트 작성 (5요소)
참가자가 Cowork 채팅에 붙여넣을 프롬프트를 같이 만든다. 반드시 5요소를 채운다:

1. **역할/직무** — "나는 ○○ 담당자입니다"
2. **입력** — 워크스페이스의 어떤 파일/형식·컬럼, 보안 제약
3. **워크플로우** — (1)…(2)…(3)… 단계 (대용량이면 분할+건수 대조 같은 안전장치 포함)
4. **출력 포맷** — 표/문서 구조, "단정 금지·근거 필수" 같은 규칙
5. **마무리 지시** — "이 흐름을 SKILL.md 1개로 정리해줘 — 트리거·입력 가정·워크플로우·출력 포맷 섹션 포함"

### 4단계 — 등록 + 보정 + 사용
1. Cowork가 SKILL.md 초안을 보여주면 본인 양식과 다른 부분만 1~2회 보정("출력에서 X 빼고 Y 추가", "내 양식 헤더 그대로 써줘", "트리거를 '○○ 돌려줘'로").
2. "이 SKILL.md를 우리 프로젝트 스킬로 등록해줘"로 등록.
3. 등록 후 "사용 예시 프롬프트"로 호출을 확인한다.

## 도출 프롬프트 5요소 (붙여넣기 골격)

```
"나는 <직군/직무> 담당자입니다. <반복 작업>을 자주 반복합니다.
입력: 워크스페이스 폴더의 <파일/형식>(컬럼: …). 사내 민감 데이터는 워크스페이스 안에서만 다룹니다.
워크플로우: (1) … (2) … (3) …  (대용량이면: 구간 분할 + 처리 건수 = 원본 건수 대조)
출력: <표/문서 구조>. 모든 판단은 '단정'이 아니라 '추가 확인 필요 후보'로 표기.
제약: 사내 데이터 외부 업로드 금지, 추측 금지, 근거 필수.
이 흐름을 SKILL.md 1개로 정리해줘 — 트리거·입력 가정·워크플로우·출력 포맷 섹션 포함."
```

## 예시 뱅크 (`references/examples/` — on-demand 로드)

직군 근접 예시 9종. 사용자 직군에 가까운 것을 1개 골라 1단계에서 보여준다(그대로 복붙 금지, 그림 잡기용).

| 파일 | 도메인 |
|---|---|
| `audit-voucher-anomaly.md` | 감사·진단 — 전표·원장 이상 스크리닝(분할 처리+누락 대조) |
| `scm-news-sensing.md` | 물류/SCM — 주간 이슈·뉴스 센싱 리포트 |
| `weekly-risk-sensing.md` | 공통 — 주간 리스크 센싱 |
| `meeting-notes-summary.md` | 사무 공통 — 회의록 → 정형 요약(결정·액션아이템) |
| `finding-to-report.md` | 공통 — 발견사항 5단(현황-기준-차이-영향-권고) 보고서 변환 |
| `source-verification.md` | 공통 — 1차 출처 검증 |
| `audit-multistep-agent.md` | 감사 — 대상사 배경 분석 멀티스텝 |
| `pii-scan-mask.md` | 정보보안 — 업로드 전 민감정보 스캔·마스킹 |
| `algo-implement-verify.md` | 공학 R&D — 알고리즘 구현+검증 |

> 감사·전표처럼 **민감 데이터**를 다루는 미니스킬은, `pii-scan-mask`로 먼저 표본을 가린 뒤 `audit-voucher-anomaly` 점검으로 가는 순서를 권한다(가리고 → 점검).

## 핸드오프 / 팁

- 아직 *무엇을* 자동화할지 자체가 막막하면 → **find-work**(itda-work)로 발굴부터.
- "뭘 굳힐지"는 정했는데 오늘 크기가 막막하면 → **hour-slice**로 한 조각부터.
- 도출은 **항상 초안**이다. 본인 양식·용어로 1~2회 보정해야 진짜 본인 스킬이 된다.
- 보안: 제작·도출 단계에서는 마스킹 표본만. 실데이터는 등록 후 사내 정책대로.
