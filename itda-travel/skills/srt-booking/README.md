# srt-booking — 동작 원리와 주의

> ⚠️ **실험적 스킬입니다. 배포하지 마세요.**
> SR 비공식 API에 의존하며 이용약관(ToS) 위반 소지가 있습니다. 개인 학습·실험
> 목적으로만 사용하고, 공개 marketplace에 배포하지 않습니다. 자세한 사용법은
> [GUIDE.md](GUIDE.md), 실행 계약은 [SKILL.md](SKILL.md) 참조.

## 무엇을 하는가

SRT(수서고속철) 열차 **검색·예약**까지만 합니다. 결제·취소는 하지 않습니다
(사용자가 SR 앱/홈페이지에서 직접). 자동 연타·취소표 매크로도 제공하지 않습니다.
KTX는 자매 스킬 [ktx-booking](../ktx-booking/SKILL.md)을 쓰세요.

## 어떻게 동작하는가 — NetFunnel 가상 대기열

SR 예매 시스템(`etk.srail.kr`)은 **NetFunnel**(가상 대기열 솔루션,
`nf.letskorail.com`)로 동시 접속 폭주를 제어합니다. 명절·인기 시간대에 뜨는
"접속자가 많아 대기 중" 화면이 바로 이것입니다.

`SRTrain` 의 `NetFunnelHelper` 는 이 **대기열 키(netfunnel key)를 프로그램적으로
생성·관리**해 대기열을 통과합니다:

1. 오피코드 흐름 `getTidchkEnter(5101)` → `chkEnter(5002)` → `setComplete(5004)`
   으로 대기열 진입·확인·완료를 처리합니다.
2. 응답 상태로 대기 여부를 판별합니다: `200`(대기 불필요·통과) · `201`(대기 필요) ·
   `502`(이미 완료).
3. 생성한 키를 캐시해 여러 요청에서 공유합니다.

즉 SR이 트래픽 제어용으로 둔 사람용 대기열을, 자동화 클라이언트가 프로그램으로
통과하는 것입니다.

## KTX(DynaPath)와의 차이

| | KTX (korail2-ncard) | SRT (SRTrain) |
|---|---|---|
| 방어 메커니즘 | **DynaPath 앱 무결성 검증** — 봇 자체를 식별·차단 | **NetFunnel 가상 대기열** — 트래픽 폭주 제어 |
| 우회 방식 | 무결성 토큰 위조(루팅/디버깅/에뮬 플래그 위장) | 대기열 키 자동 생성으로 통과 |
| 우회 강도 | 강함(앱 무결성 통째 위장) | 상대적으로 약함(대기열은 봇 식별이 주목적 아님) |
| cancel | 라이브러리 버그로 동작 불가 | 정상 동작 |

SRT는 앱 무결성 위장(루팅/디버깅 플래그 등)이 없어 KTX보다 우회 강도가 약하지만,
**여전히 비공식 자동화**입니다.

## 왜 이것이 문제인가

- **ToS 위반 소지** — SR 이용약관은 자동화 수단을 통한 접속·예약을 금지하고, 위반 시
  이용 제한을 둘 수 있습니다.
- **지속 불가능성** — SR 측 변경(대기열 프로토콜·역코드 등)으로 동작이 멈출 수 있습니다.
- **cancel** — SRTrain의 `cancel` 은 정상 동작합니다(korail2와 대조). 그러나 우리 스킬은
  취소가 비가역·고위험이고 두 스킬의 일관성을 위해 `cancel` 을 **비목표**로 둡니다
  (취소는 SR 앱/홈페이지로 안내).

## 배포 금지

이 스킬은 **개인 학습·실험 목적**입니다. 공개 배포는 ToS 위반 도구를 불특정 다수에
확산시키므로 하지 않습니다. PRIVATE 저장소에서만 관리하며, 사용에 따른 책임(계정
제재 등)은 사용자 본인에게 있습니다.
