# market-scan — 시장조사

## 이 스킬이 푸는 문제

나는 새 시장에 들어갈지 고민 중인데, "이 시장 어떤지 알려줘"라고 하면 결과가 **뉴스 클리핑**으로 돌아온다 — 기사 10개에 "신뢰도 높음/중간/낮음"이 붙은. 정작 결정에 필요한 건 시장 규모, 경쟁 구도, 누가 무슨 근거로 그 숫자를 냈는지인데, 그건 내가 다시 일일이 원문을 열어 따져야 한다. 그렇다고 긴 조사 프롬프트를 매번 외워 쓸 수도 없다.

이 스킬은 두 가지를 자동화한다.

1. **빈칸 채우기** — 내 상황만 말하면 짧은 인터뷰로 조사 목적·범위·핵심 질문을 받아낸다. 긴 프롬프트가 필요 없다.
2. **검증된 구조화** — 1차 출처 우선, 핵심 수치 교차검증, 사실/추정 분리, 발행일·링크·등급 병기를 절차로 적용해 **의사결정용 보고서 한 장**을 만든다.

## 인접 스킬과의 경계

이 스킬은 검증 엔진을 새로 만들지 않는다. **수집·구조화는 직접 하고, 엄격 검증과 본문 추출은 이미 있는 스킬에 위임**한다.

| 상황 | 맞는 스킬 |
|------|-----------|
| 외부 시장·경쟁사·산업 자료를 모아 의사결정용으로 구조화 | **market-scan** (이 스킬) |
| 버그·성능·주장의 근본 원인을 가설-반증으로 규명 | `investigate` |
| 비교표·보고서를 1차 소스 강제로 엄격 팩트체크 | `ground-check` (이 스킬이 고위험 시 **위임**) |
| 국내 공식 통계·기업재무·거시지표 정형 데이터 조회 | `itda-gov:*` (이 스킬이 **라우팅**) |
| 외부 자료를 **검색해 URL·출처를 수집**(다중 검색엔진 fan-out) | `web-search` (이 스킬이 수집 **엔진**으로 활용 — 위임 아닌 하위 수집 도구) |
| 단일 URL·블로그 본문 추출 | `web-reader` / `blog-reader` (이 스킬이 수집 **폴백**으로 활용) |
| 내가 이미 가진 매출·고객 데이터의 패턴 분석 | `itda-data:data-ask` |

## 핵심 원칙 (이 스킬의 정체성)

- **횟수가 아니라 교차검증**. "3번 순환 검색" 같은 횟수 주문은 금지 — 같은 키워드로 비슷한 자료를 반복해 찾고 "검증했다"고 착각하기 쉽다. 신뢰는 **핵심 수치·주장이 서로 다른 유형의 독립 출처에서 일치하는지**에서 온다. 일치하면 신뢰가 올라가고, 다르면 차이를 명시한다.
- **신뢰도 점수를 지어내지 않는다**. 모델은 "신뢰도 85%" 같은 보정된 확신도를 만들 능력이 없다 — 지어내면 *엄밀함의 외피를 쓴 거짓 확신*이 된다. 그래서 등급은 **출처의 유형**(사람이 재확인 가능)으로만 매기고, 반드시 한 줄 근거를 붙인다.
- **사실과 추정을 분리한다**. 확인된 수치와 추정·가설을 한 문장에 섞지 않는다. 다만 시장 규모처럼 공식 수치가 없을 때는 추정으로 메우지 말라는 뜻이 아니라, **방법·가정을 공개한 명시적 추정**으로 분리해 적는다(§ 시장 규모 — 확인과 추정의 분리).
- **국내 정형 데이터는 검색보다 공공 API**. 시장규모·산업통계·기업재무처럼 공식 출처가 있는 정형 데이터는 웹 검색보다 공공데이터 스킬이 정확하다(§ 2단계 라우팅).
- **데이터 소스는 확장형 포트폴리오, 가용한 것만 정직하게 제시**. 웹 검색이 전부가 아니다 — 정형은 공공 API(`itda-gov:*`), **주제에 맞으면 도메인 전문 스킬**(부동산 `itda-realty:*`·주식 `itda-stocks:*`·외식 `eatery-trend`·상권 `naver-place` 등)까지. 고정 목록이 아니라 **주제·환경에 따라 늘어난다** — 새 스킬은 라우팅 항목 추가로 흡수하지 본문을 다시 쓰지 않는다. 어떤 소스를 쓸지는 **인터뷰에서 사용자가 고른다**(§ 1단계 데이터 소스 선택). 단 **먼저 가용성을 점검해 실제로 쓸 수 있는 소스만 보기로 올린다** — 미연결 엔진을 가용한 척 메뉴에 올리는 "거짓 메뉴"는 금지(연극이 된다).
- **메타결정은 객관식, 맥락은 주관식**. 목적·범위 같은 닫힌 결정은 AskUserQuestion 보기로(빈 화면 막힘 방지), "어떤 시장인지·뭘 알고 싶은지" 같은 맥락은 주관식으로 받는다 — 여기서 보기를 깔면 Claude의 보기가 사용자의 본인 단어를 덮어쓴다.

## 진행 절차

### 0단계: 의도 분기 + 시작 안내 (1턴)

**먼저 두 갈래로 가른다** — 사용자가 원하는 게 (a) **빠른 개요·최근 뉴스 정리**인지, (b) **결정을 위한 시장 조사**인지. 클리핑이 틀린 게 아니라, 결정할 때만 부족한 것이다 — 그러니 결정 의도가 아닌 사람에게 풀 인터뷰를 강요하지 않는다.

- 표현이 명확하면(예: "뉴스 좀 모아줘", "요즘 무슨 일 있었는지 정리", "간단히 개요만") → **개요 모드**: 최근 소식·핵심 사실을 **링크·발행일과 함께 짧게** 정리한다. 교차검증·의사결정 합성은 생략하고, **수치는 "검증 전 참고"로 표기**(확정 사실로 단정 금지·스니펫 복사 금지). 인터뷰는 건너뛴다. 끝에 "더 깊이 따져볼까요?(시장 조사 모드)"를 한 줄 제안.
- 결정 의도가 보이거나("진입해도 될까", "규모 얼마", "경쟁사 분석해줘") **모호하면** → **시장 조사 모드**: 아래 1단계 인터뷰로 진행.

시장 조사 모드면 짧게 안내한다: 질문 2~3개로 목적·범위를 잡은 뒤 조사하며, 산출물은 출처·발행일·등급이 붙은 보고서 한 장이라는 것. 첫 메시지에서 이미 목적·범위를 충분히 말했다면 그 질문은 건너뛴다(중복 질문 금지).

### 1단계: 인터뷰 — 빈칸 채우기 (2~4턴)

**목표**: 조사 설계에 필요한 최소 정보를 받아낸다. 한 번에 다 묻지 말고 묶음으로 나눠 묻는다.

**Q1 — 대상과 맥락 (주관식)**: "어떤 일을 하시고, 어떤 시장·주제를 조사하려고 하세요?" 사용자가 "OO 시장"처럼 넓게 답하면 한 겹 좁힌다("그 안에서 특정 제품군/세그먼트가 있나요?").

**Q2 — 조사 목적 (AskUserQuestion 사용)**: 목적이 정해져야 수집 항목이 정해진다.
- 질문: "이 조사는 결국 무슨 결정에 쓰시나요?"
- 옵션(우열 없음, Recommended 금지):
  1. 신규 진입/사업화 판단
  2. 경쟁사·대체재 동향 파악
  3. 시장 규모·성장성 산정 (투자·기획용)
  4. 가격·유통·고객 이해 (영업·제품 전략)

> **목적별 핵심 수집 항목 (선택 즉시 적용)** — 이게 2단계 수집 목록과 보고서 §2 행을 좌우한다. "시장 조사"는 목적 없이는 뜬구름이다:
> - **① 진입검토** → 시장 규모·성장률 · 경쟁 구도·진입장벽 · 규제 · 수익성 단서 → 산출: go/조건부/no-go
> - **② 동향파악** → 주요 플레이어·점유율 · 신제품·가격 변화 · 최근 사건 · 대체재 → 산출: 경쟁사 비교표 + 타임라인
> - **③ 규모산정** → 공식 통계·공시(DART)·협회 · CAGR · 세그먼트 분해 → 산출: 규모(확인/추정 분리) + 추세 + 산정 근거
> - **④ 가격·유통·고객** → 가격대 · 유통 채널 · 고객 세그먼트·구매 행태 → 산출: 가격·채널 맵 + 고객 프로파일

**Q3 — 범위 (AskUserQuestion 사용, 한 번에 묶음)**:
- 지역: 국내 / 글로벌 / 국내+해외 비교
- 고객: B2B / B2C / 공공 / 혼합
- 기간: 최근 1년 / 최근 3년 추세 / 최근 5년 장기추세
- (사용자가 답하기 어려워하면 "최근 3년·국내·혼합"을 기본값으로 제안)

**조사 깊이 (자동 추론, 묻지 않음)**: 목적·표현으로 스캔 깊이를 정한다 — 이게 데이터 소스 기본 선택을 좌우한다.
- **빠른 스캔**: 동향 파악 목적, "간단히/대충 감만"류 표현 → 핵심 수치·경쟁사 3~5곳·최근 사건 중심, 교차검증은 시장 규모 등 1~2개 핵심 수치만.
- **심층 스캔**: 진입검토·투자·규모산정 목적, "깊게/철저히"류 표현 → 전 항목 교차검증 + 고위험이면 ground-check 위임(§ 검증 강도).

**Q4 — 데이터 소스 선택 (AskUserQuestion, multiSelect)**: 어떤 소스로 조사할지 사용자가 고른다. 이 단계가 "검색 전용이 아님"을 사용자에게 직접 보여준다.

**먼저 가용성을 조용히 점검**(1회)해 **실제로 쓸 수 있는 소스만 보기로 올린다** — 미연결 엔진을 가용한 척 올리는 "거짓 메뉴"는 금지:

| 소스 | 가용성 점검 | 보기 처리 |
| --- | --- | --- |
| 웹 검색·원문 확인 (내장 WebSearch) | 항상 가용(키 불요·단일 인덱스) | 항상 포함, 기본 ☑ |
| **다중엔진 웹 검색** (`web-search`) | `web-search`의 `--check-env`(또는 환경·"Claude 지침"에 `TAVILY_API_KEY`·`NAVER_SEARCH_CLIENT_ID`·`NAVER_SEARCH_CLIENT_SECRET`·`SERPER_API_KEY`·`EXA_API_KEY`·`PERPLEXITY_API_KEY` 중 **1개 이상**) 존재 확인 | **키 보유 시에만 포함**(거짓 메뉴 금지). 키 0개면 보기에서 빼고 내장 WebSearch만 — "(키 감지됨: tavily,naver…)" 표기 |
| 공공데이터 API (`itda-gov:*`) | 작업폴더 `.env`/환경변수에 관련 키(`KOSIS_API_KEY`·`DART_API_KEY`·`ECOS_API_KEY` 등) 존재 확인 | 포함(키 없어도 공식사이트 폴백 가능). "(키 감지됨)" / "(키 미설정→발급 안내)" 표기 |
| **주제 연관 전문 스킬** | **시장 주제로 후보 판별** 후 설치/가용 확인 — 부동산→`itda-realty:*`(실거래·가격지수), 주식·ETF·섹터→`itda-stocks:surge-data`, 외식·F&B→`itda-travel:eatery-trend`, 창업 상권·로컬→`naver-place`(hyve serve 연결 필요 — web_browse MCP)/`place-finder` | 주제 적합 + 가용 시 포함(예: "부동산 시장"이면 itda-realty 제안). 무관/미설치면 제외 |

- 질문: "어떤 데이터 소스로 조사할까요? (여러 개 선택 가능)"
- 기본 체크: 빠른 스캔 → [웹] + (키 있으면)[다중엔진 web-search]. 심층 스캔 → 추가로 주제 연관 전문 스킬·공공 API + (키 있으면)다중엔진 web-search를 권장 체크. 사용자가 더하거나 뺀다.
- 미가용 소스는 보기에서 빼되, 가치가 크면 한 줄 안내한다("공공데이터 키를 발급받으면 KOSIS·DART 통계를 더 정확·빠르게 끌어올 수 있습니다").

**Q5 — 이미 아는 것 / 핵심 질문 (주관식, 선택)**: "이미 알고 계신 사실이나, 특히 답을 알고 싶은 질문이 있나요?" 없으면 건너뛴다.

1단계 종료 조건: 대상 시장 + 목적 + 지역/고객/기간 + **선택된 데이터 소스**가 한 줄로 요약된다. 사용자에게 "이렇게 조사하겠습니다" 한 줄 확인 후 2단계로.

### 2단계: 선택된 소스로 수집 (조사 본체)

1단계 Q4에서 **선택된 소스 엔진으로 수집**한다. 웹은 항상 baseline, 공공 API는 정형 데이터 1순위, 주제 연관 전문 스킬은 선택됐을 때 함께 호출한다.

- **심층 스캔 시**: 트렌드·경쟁사·해외 자료 같은 비정형 다출처 조사는 WebSearch로 후보를 넓게 펼친 뒤(키워드·기간·언어를 바꿔 여러 각도로) 후보 URL을 WebFetch로 **원문 확인**한다. 검색 스니펫만 보고 옮기지 말고, 핵심 수치는 § 교차검증 규칙대로 서로 다른 유형의 출처로 대조한다.
- **출처 다양화 — `web-search` 선택 시**: 내장 WebSearch는 단일 인덱스라 출처가 한쪽으로 쏠린다. Q4에서 `web-search`가 선택됐으면(키 보유) 이를 활성화해 키 보유 엔진을 한 번에 fan-out 한다 — **국내 Naver 색인·Exa 시맨틱·Perplexity 요약**까지 겹치지 않는 인덱스를 병합해, 교차검증의 전제인 "서로 다른 유형의 독립 출처"를 넓힌다. **국내(지역=국내) 조사에서 Naver 색인은 글로벌 엔진이 못 메우는 영역**이라 특히 유효하다. `web-search`는 URL·발췌 목록만 돌려주므로, 핵심 수치는 여느 때처럼 WebFetch(막히면 `web-reader` 폴백)로 **원문 확인** 후 반영한다(스니펫 금지는 동일).
  - **비용 가드**: `web-search auto`는 키 보유 엔진을 모두 호출하고 **Perplexity·Exa는 요청당 과금**이다. 기본은 무료 엔진(`--engines tavily,naver`)으로 돌리고, 유료 엔진은 시장 규모·점유율 같은 **핵심 수치 교차검증이 꼭 필요할 때만** 범위를 넓힌다.
- **무거운 1차 자료(PDF·증권사 리포트·산업백서) 시**: 해당 자료의 URL을 WebFetch로 본문을 끌어온다. 본문이 비거나 막히면 `web-reader` 스킬로 폴백한다. 검색에 안 잡히는 비공개·유료 자료는 사용자가 원문 링크를 주거나 핵심 내용을 붙여넣으면 그대로 근거로 반영한다.
- **공공 API 선택 시**: 아래 라우팅 표대로 `itda-gov:*` 호출.
- **항상**: 핵심 수치는 원문 확인(스니펫 금지), 발행일·링크·등급 병기.

**출처 우선순위** (위가 우선, 아래는 보조):
1. 공공기관·정부 공식 통계, 규제기관 공시
2. 산업 협회·전문 리서치 기관 보고서, 기업 공시·IR
3. 주요 언론 보도
4. 블로그·홍보성 콘텐츠·출처 불명 (보조 자료로만, 단독 근거 금지)

**국내 정형 데이터 라우팅** — 아래는 웹 검색 대신 해당 스킬을 우선 호출한다.

| 필요한 것 | 우선 경로 |
| --- | --- |
| 시장 규모·산업·인구 통계 | `itda-gov:kosis` (KOSIS 국가통계) |
| 거시지표(GDP·금리·환율·물가) | `itda-gov:ecos` (한국은행 ECOS) |
| 기업 재무·직원수·사업보고서·배당·소송 | `itda-gov:dart` (전자공시 DART) |
| 정부 지원사업·창업 공고 | `itda-gov:funding` (K-Startup) |
| 공공 입찰 동향 | `itda-gov:g2b` (나라장터) |
| 부동산 실거래 | `itda-gov:realestate` |

> **정형 데이터 단위·의미 검증 (필수)**: 공공 API가 돌려준 값은 **단위와 측정 대상을 반드시 확인**한다. 표 이름에 "매출액"이 있어도 값이 "지수(점)"이면 그건 매출이 아니라 경기 체감지수다(시장규모 아님). 보고서에 단위(억원·조원·%·점·명)와 측정 대상을 명시하고, 질문과 단위가 안 맞으면 그 표를 버리고 다른 표를 찾는다. 또 검색 키워드 하나에 표가 여러 개 나오면(예: "가정간편식"→지역·라이프스타일 표) 제목만 보고 고르지 말고 내 세그먼트·질문에 맞는 표인지 본다. *(라이브 실측: KOSIS "식품산업 부문별 매출액" 표가 실제로는 현황/전망지수 '점'을 반환 — 제목만 믿으면 오답.)*

**수집 폴백 체인** — 정형 데이터 1건마다 아래 순서로 내려간다. 한 단계가 막히면 다음으로:

1. 해당 `itda-gov:*` 스킬 호출(Skill 도구, Cowork에서는 자연어로 활성화).
2. 스킬 **미설치 또는 미인증**(API 키 없음)이면 → 해당 기관 공식 사이트를 WebFetch.
3. WebFetch **실패**(본문 < 500자 · 4xx/5xx · "JavaScript required"류 안내문만)이면 → **`web-reader` 스킬로 fallback**. KOSIS·DART 같은 공공 포털은 JS 렌더링이라 맨 WebFetch가 자주 빈 본문을 준다 — 스니펫만 보고 수치를 옮기지 말 것.
4. 모두 실패하면 → 해당 항목은 "미확인"으로 남긴다(추측으로 메우지 않음). 단 시장 규모는 § 추정 절차로 명시적 추정 허용.

그 외(트렌드, 경쟁사 동향, 해외 자료)는 WebSearch → 후보 URL을 WebFetch로 **원문 확인**(실패 시 web-reader 폴백). **검색 스니펫만 보고 수치·주장을 옮기지 말 것.**

**교차검증 규칙**: 시장 규모·성장률·점유율 같은 **핵심 수치는 서로 다른 유형의 출처로 대조**한다. 수치가 다르면 둘 다 적고 차이의 원인(조사 기간·정의·표본 차이 등)을 한 줄로 설명한다.

> **단일 출처 함정 + 리포트밀 주의 (시장 수치 특유)**:
> ① 시장규모·CAGR 수치는 **한 리서치사의 원 보고서가 여러 기사에 재인용**되는 경우가 많다. 서로 다른 기사 2개가 같은 원 보고서를 인용하면 그건 독립 출처 2개가 아니다. 인용의 **원 출처(발행 기관·발행 연도)를 추적**해, 사실상 단일 출처면 "사실상 단일 출처(○○연구원 20XX)"로 표시한다.
> ② "X 시장 규모 / X market size"로 검색하면 상위에 **리포트 판매 사이트**(marketgrowthreports·wiseguyreports·verifiedmarketreports·businessresearchinsights·globalgrowthinsights 류)가 뜬다 — 산출 방식 없는 자동생성 수치가 많고 사이트마다 값이 크게 엇갈린다. **C등급**으로 보고 단독 근거 금지. 국내는 `atfis.or.kr`(식품산업통계)·부처 통계 같은 1차 출처로 대조한다. *(라이브 실측: HMR 검색 상위 8건 중 6건이 리포트밀.)*

### 2.5단계: 시장 규모 — 확인과 추정의 분리

공식 수치가 있으면 그대로 인용(등급 A/B). **없으면 추정하되 반드시 방법·가정을 공개**한다 — 시장 규모는 원래 공식 수치가 없는 경우가 많고, 여기서 "확인 안 됨"만 남기면 조사가 무의미해진다.

- **Top-down**: 상위 시장 규모 × 세그먼트 비중. (예: 전체 보험시장 규모 × 반려동물보험 추정 비중)
- **Bottom-up**: 모수 × 침투율 × 객단가(ARPU). (예: 반려동물 가구 수 × 보험 가입률 × 연 보험료)
- 두 방식 결과가 크게 다르면 **둘 다 적고** 차이 원인을 설명한다(어느 가정이 결과를 흔드는지).

추정치에는 **"(추정)" 표기 + 사용한 모수·출처·가정**을 함께 적는다. 추정을 확인된 사실처럼 쓰지 않는다. 모수 자체(가구 수·상위 시장 규모)는 1차 출처로 확인한다 — 추정의 입력까지 추측이면 결과는 근거 없는 숫자다.

### 3단계: 신뢰도 등급 부여 (출처 유형 기준)

각 핵심 자료에 등급을 매긴다. **이 등급은 모델이 만든 확신도가 아니라 출처의 유형 분류다** — 사람이 재확인할 수 있어야 한다. 도메인 판정은 `ground-check`의 1차 소스 휴리스틱과 일치시킨다(`*.go.kr`·`*.gov`·기업 공식 도메인·법령 원문 = 1차).

- **A**: 공공기관·공식 통계·기업 공시·원자료에 가까운 자료, 조사 방법·기준이 명확함
- **B**: 협회·전문 리서치 기관·주요 언론·기업 보고서 등 믿을 만한 2차 자료
- **C**: 블로그·홍보성·출처나 산출 방식이 불명확한 자료 (단독 근거 금지)

**등급만 쓰지 말고 부여 이유를 1문장 붙인다.** 예: "B — 업계 협회 보고서이나 표본 수가 표기되지 않음."

가능하면 세 축을 구분해 적는다: (1) **출처 신뢰도** — 낸 기관, (2) **데이터 신뢰도** — 산출 방식·표본·기간, (3) **활용 적합도** — 내 조사 목적/세그먼트에 직접 맞는가(예: B2B를 보는데 B2C 통계뿐이면 적합도 낮음). 세 번째 축이 시장조사의 핵심이다 — 권위 있는 통계라도 내 세그먼트와 다르면 결정 근거로는 약하다.

### 4단계: 구조화 보고서 작성 + 저장

연결 폴더 `market-scan/` 아래에 저장한다(Cowork 마운트가 있으면 `mnt/market-scan/`, 없으면 `./market-scan/`). 폴더가 없으면 만든다. 동시에 대화에도 핵심을 보여준다.

**파일명**: `YYYY-MM-DD_HHmm_<시장슬러그>.md` (예: `market-scan/2026-06-09_1530_국내반려동물보험.md`)

**보고서 템플릿** — `templates/report.md` 구조를 따른다:

```markdown
# 시장조사: <대상 시장>

작성일: <YYYY-MM-DD HH:mm>
조사 목적: <Q2 선택>
범위: 지역 <..> · 고객 <..> · 기간 <..>

## 1. 핵심 요약 (5줄 이내)

## 2. 시장 현황 표

(시장 규모 / 성장률 / 주요 고객군 / 경쟁사·대체재 / 가격·유통·채널 / 규제·정책 / 최근 1~2년 주요 사건)
```

**사실/추정 표기**: 보고서 전체에서 확인된 사실과 추정·가설은 문장 단위로 구분한다(추정은 "추정:" 또는 "(추정)" 표기). 출처 간 수치가 엇갈린 항목은 7번에 모은다.

저장 후 파일 경로를 알리고, **AskUserQuestion으로 다음 행동을 제시**한다:
1. 특정 항목(예: 경쟁사 / 규제) 더 깊이 파기
2. 핵심 수치를 ground-check로 재검증
3. 이 보고서를 슬라이드/문서로 변환 (`document-skills:pptx` / `itda-work:hwpx-reader` 등 연계)
4. 여기까지 — 보고서만 받고 종료

## 검증 강도 — 기본 vs ground-check 위임

이 스킬은 1차 소스 강제·hedge 차단·독립 검증 라운드를 **새로 정의하지 않는다**. 이미 `ground-check`가 그 절차를 갖고 있으므로, 강한 검증이 필요하면 그쪽에 위임한다(중복 방지).

- **기본 검증 (이 스킬 내장)**: 핵심 수치는 원문 WebFetch로 직접 확인(스니펫 금지) + 서로 다른 유형 출처 교차 + 사실/추정 분리 + 발행일·링크 병기.
- **ground-check 위임 (권장 상황)**: 투자·계약·진입 결정처럼 리스크가 크거나, 사용자가 "팩트체크 수준으로"·"출처 엄격하게"를 요구하거나, 심층 스캔에서 핵심 수치가 단일 출처로 의심될 때. → `ground-check` 스킬을 활성화해 핵심 수치 표를 1차 소스 강제로 재검증한다. 결과(일치/불일치/미확인)를 보고서 7번과 등급에 반영한다.

## 병렬 팬아웃 — deep-researcher 서브에이전트 위임

심층 스캔에서 독립된 소스 축이 여러 개면(예: 시장규모 / 경쟁사 / 규제 / 해외 사례), 환경에 `itda-work:deep-researcher` 서브에이전트가 있을 때 **축별로 병렬 위임**할 수 있다. 각 축의 조사 질문 + 상위 목적 + 산출 요구를 프롬프트로 넘기면, 격리 컨텍스트에서 조사하고 근거 붙은 구조화 요약만 반환한다 — 수집 원문이 본 대화를 오염시키지 않고, 축들이 동시에 진행된다. 반환 요약의 출처·발행일은 본 스킬의 기본 검증(원문 확인·교차·사실/추정 분리) 기준으로 보고서에 통합한다.

서브에이전트가 없는 환경(구형 클라이언트 등)에서는 본 컨텍스트에서 순차 진행한다 — 절차는 동일하고 병렬성만 잃는다.

## 종료 신호

"여기까지" / "그만" / "보고서로 정리해줘"가 나오면 단계와 무관하게 즉시 4단계로 점프. 미완 항목은 7번 "추가 확인 필요"에 모으거나 `미정`으로 남긴다.

## 범위 밖 (별도 스킬)

- **내 데이터 분석**(이미 가진 매출·고객 데이터의 패턴 분석)은 외부 자료 수집·해석이 아니라 보유 데이터셋 분석이므로 `itda-data:data-ask` 계열로 안내한다.
- **근본 원인 규명**(왜 느린지·이 주장이 맞는지)은 가설-반증 조사인 `investigate`로 안내한다.
- 마운트 폴더·Google Drive·Notion 등 사내/커넥터 출처 기반 조사는 이 스킬의 1차 대상이 아니다(이 스킬은 공공 웹·공공데이터 우선). 1차 소스 팩트체크가 목적이면 `ground-check`(Category A 한정)로 안내한다.
