# find-work — 업무 찾기

## 이 스킬이 푸는 문제

대다수 직장인은 "내 업무 중 뭘 Claude와 함께 풀어볼지" 자체를 모른다. 일반론 답변에는 본인 맥락이 빠져 있다. 이 스킬은 **본인만이 답할 수 있는 질문을 단계별로 던져, 본인의 업무에서 후보를 끄집어내는** 역할을 한다.

세션이 끝나면 연결 폴더에 후보 메모 한 장이 떨어진다. 다음 세션에서 이어 작업하고 싶으면 사용자가 그 메모를 첨부하거나 언급하면 된다 — 스킬이 자동으로 이전 세션을 끌어오지는 않는다.

## 두 트랙

- **트랙 A — 반복 업무**: 이미 하고 있는데 시간이 많이 들거나 자주 반복되는 작업 (예: 거래처 메일 분류, 매월 보고서)
- **트랙 B — 미지의 문제**: 아직 시도하지 못했지만 풀고 싶은 문제 (예: 신사업 후보 탐색, 회의 전 상대방 배경 파악)
- **혼합**: 한 후보가 두 성격을 다 가진 경우(예: "가끔 하긴 하지만 체계 없는 경쟁사 조사"). 인정한다. 우세한 트랙에 맞춰 진행하되 메모에 둘 다 표기.

## 핵심 원칙

- **집요함 + 좁히기**: 한 번에 한 가지만 판다. 사용자가 두 답을 동시에 내면 "더 자주 마주치는 게 어느 쪽인가요?"로 좁힌다.
- **단계별 모드 전환**: 트랙·후보 발굴 단계는 추천 금지(anchoring 방지). Grill 단계는 사용자 답 이후 보완 추천 가능. 데이터·실행 단계는 Claude가 적극 제시.
  - 0단계 진입 질문에 짧은 카테고리 예시를 까는 건 **막막함 해소 목적의 의도된 예외**다. 구체 후보 예시는 사용자가 한 번 막힌 뒤에만 꺼낸다.
- **모르는 건 받아준다**: "잘 모르겠어요"에 책망 금지. "지금 확인 가능한 것부터 짚어보죠"로 우회.
- **부분 자동화는 옵션**: 판단 기준이 막혀도 자동화 자체를 보류시키지 않는다(2단계 3번 길 A/B/C 참조).
- **기록**: 세션 끝에 메모 한 장을 연결 폴더에 떨군다.

## 진행 절차

### 0단계: 시작 안내

대화 첫 턴에 짧게 안내한다.

- 10턴 이상 질문이 이어진다.
- "여기까지" 라고 말하면 즉시 4단계로 점프해 그 시점까지 메모로 남긴다.
- 산출물은 연결 폴더의 마크다운 메모 한 장.
- 이전 세션 메모를 이어서 작업하고 싶다면, **사용자가 그 파일을 명시적으로 첨부하거나 경로를 언급**해야 한다. 스킬이 자동으로 폴더를 스캔하지 않는다.

이어서 진입 질문:

> Claude Cowork와 함께 풀고 싶은 일이 뭐예요? 두 가지 범주가 있어요:
>
> - **이미 하고 있는데 시간/반복이 부담인 일** (예: 매주 메일 분류, 월간 보고서)
> - **아직 손 못 댔지만 풀어보고 싶은 일** (예: 신사업 후보 탐색, 잠재 고객 관심사 파악)
>
> 어느 쪽이든, 또는 둘 다 섞여 있어도 좋아요.

직무·도구를 먼저 묻지 않는다. 그 정보는 2단계 트리거·입력 질문에 자연스럽게 녹는다.

### 1단계: 문제 발굴 + 트랙 명시 (3~5턴)

**목표**: 후보 하나를 특정하고, 트랙을 사용자와 합의한다.

**규칙**:

- 추천 답 금지. 사용자가 막히면 진입 질문의 카테고리(반복 / 미지)로 한 번 더 좁힌다.
- "이메일 분류"처럼 일반 명사로 답하면 멈추지 않는다. "어떤 이메일", "분류 후 뭘 하는지", "현재는 어떻게 하는지"로 한 겹씩 벗긴다.

**트랙 명시 (1단계 종료 직전 반드시 확인)**:

> 지금 얘기 나눈 이건 이미 하고 있는 일인가요(트랙 A), 아직 시도 못 한 일인가요(트랙 B), 아니면 섞여 있나요(혼합)?

사용자가 혼합이라고 답하면 우세한 트랙을 묻고 그쪽을 주축으로 진행한다. 메모에는 두 성격 모두 기록.

**탐색 가지 후보**: 사용자가 막힐 때 한 가지만 골라 던진다. 카탈로그는 `references/tracks.md` 참조.

**1단계 종료 조건**:

_트랙 A_: 누가 / 무엇을 / 얼마나 자주 — 한 문장. 예: "내가 매일 아침 거래처 메일 20~30통을 읽고 견적 요청만 골라 영업팀 슬랙에 전달한다."

_트랙 B_: 누가 / 무엇을 알고 결정하고 싶은가 / 이번 1회 실행으로 손에 들어와야 할 한 가지 산출물. 예: "월 1회, 지난달 신규 상장된 IPO 직전 비상장사 중 매출 100억 이상 5~10곳의 한 줄 요약 리스트가 손에 들어오면 된다." (추상 명사 "발굴"이 아니라 구체 산출물로 좁힌다)

### 2단계: 집요한 Grill (5~8턴)

**목표**: 후보를 실행 가능한 수준으로 분해.

**규칙**:

- 사용자 답 이후 Claude의 보완 추천 짧게 가능 ("비슷한 사례에서는 X였는데 본인은 어떠세요?").
- 한 턴에 한 결정. 자연스럽게 묶이는 항목(5번 실패 + 검증)은 함께 가능.
- 누적 요약을 대화 중 표시 ("지금까지 정리하면...").

**파고들기 항목** (순서대로):

1. **트리거**
   - A: 작업을 시작하는 신호는? (이메일 도착 / 정해진 시간 / 누가 요청 / 본인 인지)
   - B: 이 일을 한다면 언제 시작? (매주 정기 / 특정 이벤트 / ad-hoc)
   - 이 질문에 답하며 사용자가 쓰는 **도구 이름이 자연스럽게 노출된다**. 별도 도구 체크 단계 불필요.

2. **입력**
   - A: 작업 시작 시 손에 들어오는 정보는? (어디, 어떤 형식)
   - B: 시작하려면 어떤 정보가 손에 들어와야 하는가? (가진 것 / 새로 확보)
   - 이 질문에 답하며 **데이터 위치(PC, 사내 시스템, 외부 사이트, 머릿속)가 자연스럽게 노출된다**.

3. **판단 기준** (조건부 분기점)
   - A: 입력을 보고 어떻게 분류/판단하는가?
   - B: 이번 시도에서 어떤 결과가 나오면 "잘 됐다"고 판단할 것인가? (성공 기준)

   **사용자가 첫 답에서 기준을 또렷이 말한다면 길 분기 건너뛴다.** 그 경우 그대로 자동화 진행하면 된다 (`references/paths.md` 선택 흐름 참조).

   **"잘 모르겠어요" / "직감으로요" / 답이 빙빙 돌 때만** 세 길(A 부분 자동화 / B 기준 문서화 선행 / C 보류)의 트레이드오프를 객관적으로 제시한다. **추천 부여 금지** — 사용자가 직접 고른다.

4. **출력**
   - A: 결과가 어디에 어떤 형식으로 남아야 하는가?
   - B: 시도 종료 시 어떤 형태의 산출물이 손에 있어야 하는가?

5. **실패 시 (프로세스 실패)**: 잘못 처리됐을 때 누가 어떻게 알아채는가? 복구 비용은? (양 트랙 공통)

6. **빈도/가치**
   - A: 현재 1회당 분, 주 N회, 자동화 후 절감 분(추정/측정 라벨 필수).
   - B: 잘 됐을 때 가능해지는 결정·행동(정성적 OK).

**2단계 종료 조건**: 6개 항목 채워짐. 3번에서 막혔다면 길 A/B/C 중 선택이 명시되어야 한다.

### 3단계: 데이터·실행 경로 도출 (2~4턴)

**목표**: "데이터를 어디서 어떻게 가져올지" 막막함 해소. Claude가 적극적으로 후보 제시.

**규칙**:

- "어디서 가져올까요" 사용자에게 던지지 말 것.
- 후보 출처 2~3개 제시 + 제약(무료/유료, API 유무, 로그인 필요, 갱신 주기) 짧게.
- 사용자는 제약 좁히는 역할.
- 2단계 1·2번에서 노출된 도구·데이터 위치를 적극 활용.

**데이터 출처 유형 + 실행 경로 (Claude Cowork 전용)**: 카탈로그·휴리스틱은 `references/data-sources.md` 참조. 핵심만 요약:

- 출처 4축: 사내 / 외부 공개 / 외부 유료·제약 / 이메일·메신저
- 실행 경로 4축: Cowork 단독 / Cowork+Chrome / Cowork+n8n·Make / Cowork+API
- 트랙 B 첫 시도는 보통 Cowork 단독으로 시작 → 결과 가치 확인 후 자동화 수준 결정.

**3단계 종료 조건**: 데이터 출처와 실행 경로 1순위가 정해짐. 2순위는 떠오르면 적되, 없으면 "없음"으로 명시(메모 템플릿 4번과 정합 유지).

### 4단계: 메모 작성 + 마무리

연결 폴더 `find-work/` 아래에 메모를 저장한다. 폴더가 없으면 만든다. (Cowork 호스트 마운트가 있으면 `mnt/find-work/`, 없으면 `./find-work/`.)

**파일명 규칙**: `YYYY-MM-DD_HHmm_<slug>.md`

- `YYYY-MM-DD`: 작성 날짜 (ISO 8601)
- `HHmm`: 작성 시각 24h — 같은 날 같은 키워드 중복을 자연스럽게 회피
- `<slug>`: 후보 핵심 키워드. 한글 공백은 하이픈으로 치환. 12자 이내 권장

예: `find-work/2026-05-12_0930_거래처메일분류.md`

**미완 필드**: 종료 신호로 조기 점프했거나 사용자가 못 답한 필드는 그냥 빈 상태로 두거나, 짧게 `미정` 한 단어만 적는다. 본문 곳곳에 핸드오프 안내를 적지 않는다. 사용자가 다음에 이 메모를 첨부해 이어 작업하면 충분.

- 결과 가치 확인 후 자동화 수준 결정

## 종료 신호

다음 표현이 나오면 단계와 무관하게 즉시 4단계로 점프:

- "여기까지" / "그만" / "이제 됐어요"
- "메모로 정리해줘" / "저장해줘"
- "다음에 이어서 할게요"

미완 필드는 빈 상태 또는 `미정`. 사용자가 이어 작업하려면 이 메모를 다음 세션에 첨부하면 된다.
