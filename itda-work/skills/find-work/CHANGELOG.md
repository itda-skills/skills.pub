# Changelog — find-work

## [0.12.0] — 2026-07-22

### New Features (#1225 — ax-proposal 설계 역수입)

- **솔루션 선행 입력의 역파기** (핵심 원칙): 사용자가 솔루션부터 말하면("~하는 시스템 만들고 싶어요")
  그대로 후보로 받아 적지 않고 "그게 되면 어떤 불편이 사라지나요?"로 밑의 문제부터 역으로 파낸다.
- **AI 적합성 3문 스크린** (1단계, 트랙 명시 전): ①입력이 문서·데이터·텍스트 ②판단 기준 언어화 가능
  ③사람이 반복 수행 — 둘 이상 "아니오"면 솔직하게 부적합을 말하고 다른 후보로. 길 C(보류)보다
  이른 시점의 정직 게이트.
- **pain 서술 컨펌** (1단계 종료 직전): 종료 조건 한 문장을 1인칭·구체 상황·정량 포함 1~3문장으로
  다듬어 명시 컨펌 (`design-from-pain.md` [HARD] 정합).
- **정량화 4축** (2단계 6번): 기존 시간·빈도에 수행 인원수·실수 시 비용 추가, 복합 빈도 연 N회 환산,
  파생 수치 산출식 괄호 병기("월 6시간 절감(추정: 2명 × 3시간)"). 메모 템플릿 §6 동반 갱신.
- **동일 세션 체이닝 안내** (4단계): 메모 저장 직후라면 파일 첨부 없이 "이어서 계획 세워줘"만으로
  plan-work가 대화 컨텍스트를 이어받는 경로 명시 — 세션 경계+첨부 마찰 제거.

### Why

경진대회 신청서 스킬 ax-proposal(itda-ceragem, 2026-07-19)과의 비교 검토(#1225)에서 도출.
같은 발굴 인터뷰 코어를 더 적은 분량으로 더 날카롭게 처리하는 게이트들(역파기·적합성 스크린·
pain 컨펌·산출식 병기)을 역수입. 객관식/주관식 분리·실측/추정 라벨 등 기존 자산은 불변.

## [0.11.0] — 2026-05-21

### New Features

- **plan-work 자매 스킬 안내 추가** (SPEC-PLAN-WORK-001 Planned 연계): "4단계: 메모 작성 + 마무리" 트랙별 다음 행동 4지선다 직후에 한 문단 추가 — 요구사항이 충분히 명확해진 사용자가 find-work 메모를 들고 plan-work 스킬로 자연스럽게 이어갈 수 있도록 출구 안내. GUIDE.md "제한사항" 영역에도 사용자 시점 한 줄 동반 추가.

### Why

find-work는 "막막함 → 후보 발굴"을 다루지만, 메모가 떨어진 직후 "그래서 다음에 뭐라고 해야 하나" 단계가 비어있었음. 본 안내는 자매 스킬 plan-work가 그 갭을 메우는 다음 단계임을 명시한다. plan-work는 SPEC-PLAN-WORK-001 Planned 상태이며, 본 안내는 구현 완료 시 본격화될 예정.

## [0.10.0] — 2026-05-13

### New Features

- **AskUserQuestion 객관식 가이드 도입**: 비개발자 사용자가 빈 답변창 앞에서 막히는 문제 해소. 메타 결정(트랙 분류, 길 선택, 출처·실행 경로 채택, 다음 행동)은 객관식으로 전환하고, 맥락 추출(후보·트리거·입력·출력)은 주관식 유지하는 분리 기준을 핵심 원칙에 명문화.
- **객관식 적용 지점 6곳 명시**:
  - 0단계 진입: 트랙 1차 분류 4지선다 (A / B / 혼합 / 잘 모르겠다)
  - 1단계 종료: 트랙 확정 3지선다 (+ 혼합 시 follow-up 2지선다)
  - 2단계 3번: 판단 기준 막힘 fallback — 길 A/B/C 3지선다
  - 2단계 6번: 시간 추정 라벨 명확화 — 실측 / 추정 2지선다
  - 3단계: 출처 후보 채택 + 실행 경로 4지선다 (Cowork 단독 / +Chrome / +n8n·Make / +API)
  - 4단계: 다음 행동 4지선다 (트랙별 옵션 분기, "지금은 여기까지" 포함)

### Improvements

- 옵션 description에 적합 상황·진입 난이도 한 줄씩 부여 지시 — 사용자가 보기만 보고 결정 가능하도록.
- 트랙 옵션에는 Recommended 표기 금지 (트랙 자체에는 우열 없음). 길 A/B/C도 동일.

### Why

비개발자 사용자의 빈 답변창 막힘 문제. 메타 결정은 의미공간이 닫혀있어 객관식 anchoring 부작용이 작고, 맥락 추출은 의미공간이 열려있어 객관식이 사용자 본인 단어를 덮어쓰는 위험이 큼. 두 경우를 분리 가이드.

## [0.9.1] — 2026-05-13

### Bug Fixes

두 트랙 dry-run 검증(트랙 A 12턴, 트랙 B 17턴)에서 확인된 결함 4건 수정.

- **2단계 3번 길 분기 트리거 조건 명시**: "사용자가 막히면"의 정의가 모호해, 판단 기준이 명확한 케이스(트랙 A 사례)에서도 길 A/B/C 트레이드오프가 불필요하게 등장할 위험이 있었다. **"잘 모르겠어요" / "직감으로요" / 답이 빙빙 돌 때만** 길 분기로 들어가도록 조건 명시.
- **3단계 종료 조건 ↔ 메모 템플릿 4번 정합 정렬**: 종료 조건은 "1순위만" 요구했는데 템플릿은 "1순위+2순위" 두 줄 강제 → 인터뷰 자연 흐름과 메모 작성이 어긋남. 종료 조건에 "2순위는 떠오르면 적되 없으면 '없음' 명시", 템플릿 2순위 줄에 `(선택, 없으면 "없음")` 라벨 추가.
- **혼합 트랙 메모 1번 정보 손실 방지**: "빈도(A) / 시작 시점(B)"과 "현재 소요 시간(A) / 못 하고 있는 이유(B)"가 한 줄로 묶여 있어 혼합 트랙에서 A·B 두 사실이 한 줄에 욱여넣어짐(실측 확인). 혼합 시 sub-bullet으로 A 측면 / B 측면 분리 작성 지시 추가.
- **4단계 다음 행동 후보 트랙별 분기**: 기존 디폴트 3개가 트랙 B 위주("프로토타입 만들기" 등)라 트랙 A의 핵심 다음 행동인 **ROI 베이스라인 측정**이 빠져 있었다. 트랙 A/B 각각 3개씩으로 분기.

### Verification

- `.validation-trackA/` (트랙 A, 김민호 대리 페르소나, 12턴, 주간 경쟁사 모니터링)
- 두 dry-run 모두 풀 사이클(0~4단계) 완주 및 메모 정상 산출 확인
