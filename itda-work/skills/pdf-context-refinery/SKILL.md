# PDF Context Refinery

## 이 스킬이 하는 일

PDF 원문 추출은 깨진 결과물을 만든다: 띄어쓰기 누락, 테이블 파편화, 문장 단절, 구조 부재. 이 스킬은 그것을 사람이 읽고 LLM이 참조할 수 있는 깨끗한 구조화된 마크다운으로 재구성한다.

핵심 문제는 *추출*이 아니라 *복원*이다. 깨진 띄어쓰기 재조합, 줄 조각에서 테이블 레이아웃 인식, 서식(이미지가 나은 경우)과 텍스트(마크다운이 나은 경우) 구분 — 이것이 이 스킬이 해결하는 문제다.

## Prerequisites

시스템 도구 (poppler-utils) — `pdftotext`, `pdfinfo`, `pdftoppm` 제공:

- **Claude Cowork**: 기본 설치됨 (poppler-utils 22.02.0, Ubuntu 22.04). 추가 설치 불필요.
- **Ubuntu/Debian**: `apt-get install -y poppler-utils`
- **macOS**: `brew install poppler`
- **Windows**: `choco install poppler`

선택사항:
- `tabula-py` (복잡한 테이블 추출 폴백): `uv pip install --system tabula-py`

> **스캔 PDF (이미지 전용)**: tesseract-ocr이 아닌 Claude의 비전 기능을 사용한다.
> `pdftoppm`으로 페이지를 PNG로 변환한 뒤, Claude가 이미지를 직접 읽고 텍스트를 재구성한다.

## Pipeline

```
PDF → Analyze → Plan → Extract → Transform → Assemble → Verify
```

각 단계를 순서대로 따른다.

## Step 1: Analyze

추출 전에 문서를 파악한다.

```bash
pdfinfo "<pdf_path>" | grep Pages
pdftotext -f 1 -l 3 "<pdf_path>" .itda-skills/sample.txt
wc -c .itda-skills/sample.txt
pdftotext -f 1 -l 10 -layout "<pdf_path>" .itda-skills/toc.txt
```

> 임시 파일은 CWD 기준 `.itda-skills/`에 저장한다 (Cowork 컨테이너 호환).

파악할 4가지:
- **언어** — 한국어, 영어, 일본어, 혼합? 띄어쓰기 규칙이 달라진다.
- **문서 유형** — 교과서, 매뉴얼, 보고서, 서식 위주?
- **총 페이지 수** — 청킹 전략을 결정한다.
- **목차 유무** — 있으면 섹션 경계 정의에 활용한다.

`sample.txt`가 거의 0바이트면 스캔(이미지 전용) PDF다. `pdftoppm`으로 페이지를 PNG 이미지로 변환한 뒤, Claude 비전으로 직접 읽는다.

## Step 2: Plan

### 사용자 확인

진행 전에 확인:
- **출력 모드**: 섹션별 파일 또는 단일 파일?
- **이미지 임베딩**: 서식 페이지를 PNG로 렌더링?

### 청킹 전략

가장 중요한 계획 결정. 한 번에 너무 많은 페이지를 처리하면 내용 손실이 발생한다 — 벤치마크에서 가장 흔한 실패 모드.

| 총 페이지 | 청크 크기 | 이유 |
|-----------|----------|------|
| < 30 | 전체 | 한 번에 처리 가능 |
| 30–80 | 30–40 페이지 | 섹션당 한 패스 |
| 80–200 | 40–50 페이지 | 병렬 패스 |
| 200+ | 50 페이지 최대 | 반드시 병렬화 |

섹션이 50페이지를 초과하면 자연스러운 소절 경계에서 분할하되, 5페이지 오버랩을 둔다.

### 병렬 실행

독립적인 섹션은 병렬 실행 가능. 대부분의 교과서 섹션은 독립적이다. 종속성이 없는 작업은 단일 메시지에서 여러 에이전트를 동시 실행한다.

## Step 3: Extract

`pdftotext -layout`으로 공간 배치를 보존한다 (테이블에 도움됨).

```bash
mkdir -p .itda-skills
pdftotext -f <start> -l <end> -layout "<pdf_path>" .itda-skills/section_raw.txt
```

이미지 임베딩이 필요한 서식 페이지:
```bash
pdftoppm -f <page> -l <page> -png -r 200 "<pdf_path>" .itda-skills/page
```

## Step 4: Transform

핵심 단계. 원시 텍스트를 읽고 구조화된 마크다운으로 재작성한다.

### 4a. 헤더

모든 출력 파일은 이것으로 시작:

```markdown
# [섹션 제목]

> 출처: [문서 제목] ([저자]), p.[시작]~[끝]

---
```

### 4b. 제목 계층

| 문서 요소 | 마크다운 |
|----------|---------|
| 대단원 | `##` |
| 절/주제 | `###` |
| 소절 | `####` |
| 본문 내 번호 항목 | `**(1)**` 인라인 |

섹션 경계에 페이지 마커 추가: `<!-- p.42 -->`

### 4c. OCR 아티팩트 제거

원시 OCR 텍스트에는 반드시 제거해야 할 아티팩트가 있다.

**페이지 머리글/꼬리글 혼입** — `84  2026 법인세 신고 핵심실무`나 `Ⅱ. 법인세 결산과 신고  95` 같은 줄은 본문이 아닌 페이지 머리글이다. 제거한다.

**글머리 아티팩트** — OCR이 `- .`이나 `. ` 접두사를 생성하는 경우가 있다. 표준 마크다운 목록 문법으로 정규화한다.

**괄호 간격** — `( 4 )` → `(4)`, `( 조특법 제32조 )` → `(조특법 제32조)`.

**고아 조각** — 짧은 줄(< 10자)이 테이블 셀 잔해이거나 머리글 조각인 경우. 적절한 문맥에 병합하거나 페이지 아티팩트면 제거한다.

### 4d. 한국어 텍스트 정리

한국어 OCR은 띄어쓰기를 완전히 제거하는 경우가 많다. 한국어 문서의 가장 큰 품질 문제다.

**띄어쓰기 규칙:**
- 조사(은/는/이/가/을/를/에/의/로/와/과/도/만/까지)는 앞 단어에 붙이고, 뒤에 공백을 둔다.
- 문장 중간에서 끝나는 줄(마침표, 쉼표, 목록 마커 없음)은 병합한다.
- 구두점 정규화: `。→ .`, 마침표 뒤 일관된 간격.

**전/후 예시:**
- 원시: `재고자산이라함은손익계산서의매출원가`
- 정리: `재고자산이라 함은 손익계산서의 매출원가`

영어: 줄 바꿈에 걸친 하이픈 단어를 재결합한다.

### 4e. 테이블 복원

두 번째로 어려운 문제. OCR이 테이블 셀을 여러 줄에 흩뜨린다.

**감지 신호:**
- 큰 공백으로 분리된 다중 열
- 반복되는 행 패턴
- 헤더 텍스트 뒤 정렬된 데이터 행

**표 형식 데이터가 감지되면 반드시 마크다운 테이블을 생성한다:**
```markdown
| 열 A | 열 B | 열 C |
|------|------|------|
| 데이터 1 | 데이터 2 | 데이터 3 |
```

구조가 모호하면 깨진 테이블보다 인용문이나 들여쓴 목록을 사용한다. 단, 테이블 생성 쪽으로 기울인다 — 벤치마크에서 테이블 누락이 핵심 실패 모드였다.

### 4f. 특수 콘텐츠

- **인용문** (`>`) — 참고사항, 예외, 법률 인용
- **계산 예시** — 명확한 들여쓰기로 단계별 표시
- **목록** — `-`로 글머리, 적절한 중첩. 파편화된 OCR 목록 항목을 완전한 문장으로 병합
- **법령 참조** — 정확한 인용 형식 보존 (조특법 제7조, 법인령 72④)

### 4g. 서식 페이지 (이미지 모드 활성화 시)

서식/템플릿 위주 페이지 → 이미지로 렌더링.

감지: 빈 셀이 있는 밀집 그리드, 사전 인쇄된 필드 라벨, 공인 도장, 매우 낮은 텍스트 밀도.

```markdown
### [서식 이름] (서식)

![서식 이름](images/p0XXX.png)

**주요 필드**: [간단한 설명]
```

## Step 5: Assemble

**단일 파일 모드** — 상단에 목차가 있는 하나의 .md 파일.

**다중 파일 모드:**
```
output_dir/
├── 00_섹션_이름.md
├── 01_섹션_이름.md
├── images/
│   └── p0042.png
└── INDEX.md
```

이름 규칙: `{NN}_{설명_이름}.md` — 번호로 정렬, 이름으로 탐색.

## Step 6: Verify

번들된 품질 검사 스크립트를 실행한다:

```bash
python ${CLAUDE_SKILL_DIR}/scripts/verify_quality.py <output_dir_or_file> --pages <total_pages>
```

스크립트가 검사하는 항목:

**라인 수 비율** — 출력 줄 수 대 소스 페이지:

| 문서 유형 | 건강 범위 | 내용 손실 임계값 |
|----------|---------|---------------|
| 밀집 교과서 | 15–45줄/페이지 | < 8줄/페이지 |
| 기술 매뉴얼 | 10–20 | < 7 |
| 서식 위주 | 3–8 | < 2 |

**구조 존재** — `##` 헤더, `###` 서브헤더, `|` 테이블 행, `![` 이미지 수. 소스에 표 데이터가 있는데 테이블이 0개 → 재처리 대상.

**스팟 체크** — 각 파일의 처음과 마지막 30줄을 읽는다. 확인: 헤더 형식 정상, 띄어쓰기 자연스러움, OCR 아티팩트 없음, 내용 중단 없음.

검증 실패 섹션은 더 작은 청크로 재처리한다 (페이지 범위를 반으로 줄인다).

## Failure Recovery

1. **내용이 너무 짧음** — 청크 크기를 반으로 줄여 재추출
2. **테이블 깨짐** — `-layout` 없이 시도하거나, 테이블 집중 페이지에 `tabula-py` 사용
3. **인코딩 문제** — EUC-KR/CP949 확인; `iconv -f CP949 -t UTF-8`로 변환
4. **스캔 PDF** — `pdftoppm -png -r 300`으로 이미지 변환 후 Claude 비전으로 직접 읽기

## Evaluation

벤치마크 검증 케이스 (한국 세무/회계 교재 기준):

### 케이스 1: 법 구조 + 테이블 복원 (16페이지)

- **프롬프트**: "이 PDF 교재의 p.85~100을 마크다운으로 변환해줘. 표와 구조를 잘 살려서."
- **기대 결과**: 법인세 결산 섹션이 깨끗한 마크다운으로 변환. 비교표가 markdown table로 재구성.
- **검증 기준**:
  - 마크다운 헤더 3개 이상 (`##`)
  - 테이블 구문 존재 (`|` 행)
  - 한국어 띄어쓰기 자연스러움
  - 최소 100줄 출력
  - OCR 아티팩트 없음

### 케이스 2: 목록 + 테이블 집중 (21페이지)

- **프롬프트**: "이 PDF의 p.325~345를 마크다운으로 변환해줘. 업종 목록, 감면율 표 정리."
- **기대 결과**: 조특법 세액감면 섹션 구조화. 업종 분류 목록이 깔끔한 리스트, 감면율 비교가 표.
- **검증 기준**:
  - 마크다운 헤더 3개 이상
  - 테이블 구문 존재
  - 목록 형식 존재 (`-` 또는 숫자)
  - 최소 120줄 출력
