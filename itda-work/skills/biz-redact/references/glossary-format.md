# 용어집(glossary.json) 형식 정본

`biz-redact`가 무엇을 가릴지 정하는 유일한 진실 소스는 **사용자 용어집**이다. 도구는 정책 판단을 하지 않고 여기 등재된 항목만 결정론적으로 치환한다. 이 문서는 용어집 파일의 스키마·제약·설계 트레이드오프를 정의한다. 실제 동작의 단일 진실원은 `scripts/biz_redact.py`이며, 본 문서는 형식과 그 근거를 설명한다.

> ⚠️ **용어집은 평문 기밀 파일이다.** 원값(거래처명·단가 등)을 그대로 담으므로 **저장소에 커밋하지 않는다**(SKILL.md [HARD] 3). 에이전트는 용어집을 Read 하지 않는다 — 사용자가 로컬에서 작성한다. 저장소에 두는 것은 합성 예시 `glossary-template.json`뿐이다.

---

## 스키마

```json
{
  "schema_version": "1.0",
  "name": "설비팀-용어집",
  "entries": [
    {"category": "거래처", "value": "현대엘리베이터", "aliases": ["현대엘리", "HYUNDAI ELEVATOR"]},
    {"category": "프로젝트", "value": "PJ-2026-047"},
    {"category": "단가", "value": "1,850,000원"}
  ]
}
```

| 키 | 필수 | 의미 |
|---|---|---|
| `schema_version` | ✅ | 스키마 버전 문자열(현재 `"1.0"`) |
| `name` | ✅ | 용어집 이름. **비기밀 라벨**(리포트·감사 로그에 노출) — 기밀을 담지 않는다 |
| `entries` | ✅ | 등재 항목 배열 |
| `entries[].category` | ✅ | 항목 종류(자유 정의, 아래 제약) — 토큰·리포트에 노출되는 **비기밀 라벨** |
| `entries[].value` | ✅ | 대표 표기. 복원 시 이 값으로 되돌린다 |
| `entries[].aliases` | 선택 | 같은 실체의 이형 표기 목록(전부 같은 토큰으로 치환) |

## category·name — 비기밀 라벨 규칙

`category`는 토큰(`⟦거래처_1⟧`)·리포트·감사 로그에, `name`은 리포트·감사 로그에 **그대로 노출**된다. 따라서 이 두 라벨 자체가 기밀을 드러내면 안 된다.

- **한국어 허용**, category는 1~24자.
- **금지 문자**(category): 공백, `⟦`, `⟧`(토큰 경계 문자).
- **비기밀 제약**: 라벨(category·name)이 어떤 entry의 value·alias와 겹치면 안 된다. `category: "현대엘리베이터"`처럼 거래처명 자체를 카테고리로 쓰면 그 라벨이 리포트로 새어 마스킹이 무의미해진다. category는 "거래처"·"프로젝트"·"단가"처럼 **종류**를 가리키는 일반 라벨이어야 한다.

### 라벨 겹침 판정 — 3단계 정책 (BLOCK / WARN / PASS)

비교는 매칭과 같은 기준(NFC + 대소문자 무시)으로 하되, "실제 유출"과 "우연한 대소문자 겹침"을 구분해 짧은 ASCII 라벨(`IT`·`PO`·`PJ` 등)의 사용성을 지킨다.

| 관계 | 처리 | 예 |
|---|---|---|
| 라벨 == 기밀(대소문자 무시) | **BLOCK** (로드 에러) | category `HDEL` + value `hdel` — 토큰 `⟦HDEL_1⟧`이 기밀을 그대로 노출 |
| 기밀 전체가 라벨에 포함(대소문자 무시) | **BLOCK** | value `PJ`가 category `PJ담당`에 통째로 들어감 — 라벨 경유로 기밀 전체 유출 |
| 라벨이 기밀 속 부분문자열 — **원문(대소문자 구분) 일치** | **BLOCK** | category `현대` ⊂ value `현대엘리베이터` — 토큰 `⟦현대_1⟧`이 기밀 앞부분을 노출 |
| 라벨이 기밀 속 부분문자열 — **대소문자 무시로만 겹침** | **WARN**(stderr, 차단 안 함) | category `IT` ⊂ `digital전환계획`의 `it` — 원문은 `IT`≠`it`, 우연한 겹침 |
| 겹침 없음 | **PASS** | category `IT` + value `홍길동전자` |

WARN은 "혹시 라벨이 기밀을 드러내지 않는지" 육안 확인을 권하는 신호일 뿐 로드를 막지 않는다. 실제로 라벨이 기밀을 노출하는 세 경우(동일·기밀⊆라벨·원문 부분문자열)는 전부 BLOCK으로 막힌다.

## value·alias — 표면형 규칙

`value`와 `aliases`의 각 문자열을 **표면형**이라 한다. 문서에서 이 표면형을 찾아 토큰으로 치환한다.

- **최소 길이 2** — 1자 표면형은 로드 에러(일반 문자 오폭이 걷잡을 수 없다).
- **중복 금지** — 같은 문자열이 entry 간·entry 내에 이중 등재되면 로드 에러(어느 토큰으로 갈지 모호 → 결정론 붕괴).
- **토큰 패턴 금지** — 표면형에 `⟦…⟧` 형태가 들어가면 로드 에러.
- **매칭 규칙**:
  - 한국어·기본: 정확 부분문자열 매칭. 조사가 붙어도 정상 치환된다("현대엘리베이터를" → "⟦거래처_1⟧를").
  - ASCII 알파벳 표면형: **대소문자 무시** + **단어 경계** 적용("HDEL"이 "SHDELL" 내부에 오매칭되지 않는다).

### 짧은 표면형 오폭 트레이드오프

**단어 경계가 적용되지 않는 짧은(2~3자) 표면형**은 일반 텍스트와 충돌할 위험이 있다. 두 부류다.

- **짧은 한국어**(예: alias `"현대"`) — "현대 사회", "현대적" 등을 잘못 가린다.
- **짧은 순수 숫자·기호**(예: value `"850"`) — `1,850,000` 안의 `850`처럼 **더 긴 값 내부에 substring 오폭**한다. ASCII 알파벳 표면형(`"HDEL"` 등)은 단어 경계가 적용돼 이 위험이 없지만, 숫자·기호만인 표면형은 경계 없이 그대로 부분 매칭된다.

도구는 이 두 부류를 **차단하지 않고 WARN(stderr)** 으로만 알린다 — 사용자 선택을 존중한다.

- **이득**: 문서에서 축약형·짧은 코드로 쓰인 항목도 잡는다(재현율↑).
- **위험**: 동음 일반 단어·더 긴 숫자 내부를 과잉 마스킹한다(정밀도↓).
- **과마스킹은 안전 방향**: 잔존(가려야 할 게 안 가려짐)보다 과마스킹(안 가려도 될 게 가려짐)이 유출 관점에서 안전하다. 그래서 도구는 차단이 아니라 WARN으로 두고 사용자 판단에 맡긴다.
- **권장**: 고유성이 높은 표면형을 우선 등재하고, 짧은 축약형·짧은 숫자는 실제 문서에서 그 표기가 쓰이는 것을 확인한 뒤 추가한다. WARN이 뜨면 마스킹본을 육안 확인해 오폭 여부를 점검한다. 금액은 `"1,850,000원"`처럼 단위·구분자를 포함한 긴 표기로 등재하면 substring 오폭을 피할 수 있다.

> **유니코드 정규화(NFC)**: 입력 문서와 표면형은 매칭 전 NFC로 정규화된다. macOS 등에서 흔한 NFD(분해형) 한글 문서에서 등재 기밀이 조용히 안 가려지던(위음성) 것을 막기 위함이다(안전 우선 — 기밀 마스킹 > 원문 바이트 보존). 원문이 NFD였으면 리포트 `normalized: true`로 표기되고, 바이트 수준 왕복 무손실은 NFC 문서 기준이다(SKILL.md 한계 참조).

## 토큰 규칙과 문서 간 일관성

- **토큰 형식**: `⟦{category}_{n}⟧` — `n`은 **카테고리 내 entry 정의 순번**(1부터). 예: 거래처 카테고리의 첫 entry는 `⟦거래처_1⟧`, 둘째는 `⟦거래처_2⟧`.
- **용어집이 순번의 진실 소스** — 토큰 번호는 문서 내용이 아니라 **용어집 정의 순서**로 정해진다. 따라서 **같은 용어집을 쓰는 모든 문서에서 같은 실체는 항상 같은 토큰**이다. 지난달 검토서와 이번달 검토서에서 "현대엘리베이터"가 둘 다 `⟦거래처_1⟧`이 되어 문서 간 비교가 성립한다(수동 마스킹의 "문서마다 치환이 달라짐" 문제 해소).
- **팀 용어집 재사용**: 회사·팀 단위로 용어집 하나를 유지하면 구성원·문서 전체에서 토큰이 일관된다. 새 항목은 배열 **끝에 추가**해 기존 토큰 번호를 흔들지 않는다.

> 이중꺾쇠 `⟦⟧`(U+27E6/U+27E7)는 LLM이 잘 보존하는 것이 실증된 브래킷이다. 다만 한국어 카테고리 라벨을 그 안에 넣는 것은 확대 적용이라, LLM이 라벨을 변형할 여지가 있다. 변형이 일어나면 `restore`가 exit 1로 안전 실패시킨다(SKILL.md 한계 참조).

## 무엇을 등재할까 — 마스킹 범위 결정

용어집에 넣은 항목은 AI가 볼 수 없다. **식별자(거래처·프로젝트코드·담당자)는 등재해 가리고, 검토 대상 수치(단가·원가율)는 보통 남긴다** — 그래야 AI가 실질 검토를 한다. 회사 정책상 수치도 금지면 단가를 등재하되, 그 경우 AI는 토큰 간 상대 비교만 가능하다. 상세는 SKILL.md "마스킹 범위 ↔ AI 수행력" 절을 따른다.

## 겹치는 표면형과 파편 잔존 (한계)

등재 표면형끼리 **포함 관계로 겹치면**(예: "거래처지원"과 "지원"을 둘 다 등재), 마스킹은 길이 내림차순으로 긴 쪽("거래처지원")을 먼저 토큰화한다. 이때 짧은 쪽("지원")의 출현이 긴 쪽 안에 흡수돼, 문서에 따라 **완결되지 않은 파편**(예: 앞이 잘린 "원" 한 글자)이 남을 수 있다. 잔존 스캔은 "완결 표면형의 미치환"을 검출하지, 부분문자열 파편까지 재조립해 잡지는 않으므로, 이런 파편은 `residual_count`에 0으로 보일 수 있다.

- 이는 결정론 substring 치환의 구조적 한계다(과잉 재조립은 오히려 오폭을 낳는다).
- **권장**: 표면형이 서로 포함 관계가 되지 않게 용어집을 정리한다(겹치는 것은 더 긴 형태 하나로 통일하거나, 파편이 기밀이면 그 파편도 별도 entry로 등재). 겹침이 불가피하면 마스킹본을 육안 확인한다.

## 로드 에러 요약 (exit 2)

다음은 전부 스키마 위반으로 로드가 거부된다(조용히 넘어가지 않는다):

- 필수 키 누락·타입 불일치, `category`·`value` 비어 있음, `schema_version`이 `"1.0"`이 아님
- `name`·`category` 비기밀 제약 위반(value/alias와 동일·기밀⊆라벨·원문 부분문자열 겹침 — 위 "라벨 겹침 3단계 정책"의 BLOCK. 대소문자 무시로만 겹치는 우연은 WARN이며 로드는 통과)
- `category` 문자 위반(공백·`⟦`·`⟧`·길이 1~24 초과)
- 표면형 최소 길이(2) 미달
- 표면형 중복(entry 간·entry 내)
- 표면형에 토큰 패턴 문자(`⟦`·`⟧`) 포함, 또는 표면형이 **생성 토큰의 부분문자열**(예: `_1` — 토큰 `⟦거래처_1⟧` 내부에 은닉돼 잔존 검사를 회피)
- 카테고리당 entry 수가 상한(9999)을 초과(토큰 번호 4자리 한계)

> **검증은 매칭과 같은 기준(NFC + 대소문자 무시)으로 판정한다.** 매칭이 유니코드 NFC 정규화 + ASCII 대소문자 무시로 이뤄지므로, 위 제약(중복·비기밀·최소 길이·부분문자열)도 같은 기준으로 검사한다. 즉 `HDEL`과 `hdel`, NFC 표기와 NFD 표기는 **같은 것으로 취급**돼 이중 등재·비기밀 우회가 로드 단계에서 거부된다(예: category `HDEL` + value `hdel`, NFD로 쓴 1글자 표면형, NFD category가 NFC value를 포함하는 경우). "검사는 통과하는데 마스킹은 가로채는" 불일치를 원천 차단한다.
