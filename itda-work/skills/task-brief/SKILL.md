# task-brief — 던지기 전 3요소 브리핑

## 이 스킬이 푸는 문제

에이전트는 집요하고 철저하게 일합니다. 그래서 **모호한 요청**을 그대로 던지면 멈출 지점을
못 찾고 엉뚱한 파일까지 헤매며 시간·토큰을 태웁니다. 에이전트가 유능할수록 명세의 빈틈을
"알아서 채우다" 원치 않은 곳으로 갑니다. 이 스킬은 요청을 던지기 **직전** 30초~2분 안에
빈틈을 메운 한 장짜리 계약 — **작업 브리프** — 으로 바꿉니다.

## 3요소 (브리프가 반드시 답해야 할 것)

| 요소 | 질문 | 없으면 생기는 실패 |
|---|---|---|
| ① **작업 범위(sandbox)** | 어디까지 건드려도 되고 어디는 안 되나? | 경계 없는 sandbox → 인접 파일·작업까지 번져 나감 |
| ② **검증 방법** | 무엇을 돌려/봐서 됐다고 판정하나? | 자기보고("확인했다")로 흘려보냄 → 거짓 성공 |
| ③ **완료 정의(DoD)** | 끝났을 때의 관측 가능한 상태는? | 종료 조건 부재 → 언제 멈출지 몰라 무한 개선 |

세 요소는 각각 **에이전트의 세 가지 폭주 유형**을 막습니다: 범위는 *번짐*, 검증은
*거짓 성공*, 완료는 *무한 루프*. 그래서 하나라도 비면 브리프로서 미완입니다.

핵심은 ② **검증 방법**입니다. 검증은 **브리프를 받는 쪽이 스스로 돌려 참·거짓을 판정할 수
있는 재현 가능한 명령·파일·수치**여야 합니다. "잘 된다"·"문제없다"·"확인했다"는 작성자의
*주장*이지 받는 쪽이 재현할 수 없습니다 — 이것이 거짓 성공이 새는 통로입니다.

## 모드 분기 (먼저 판별)

| 신호 | 모드 |
|---|---|
| 사용자가 요청 초안 텍스트를 줌 / "이 요청 다듬어줘" | **채점** |
| 목표만 있고 초안 없음 / "브리프 짜줘" | **인터뷰** |

## 발동하지 않는 경우 (과잉 발동 방지)

자명한 한 줄 요청(오타 수정·상수 하나 변경·파일 하나 이름 바꾸기 등)에는 발동하지 않습니다 —
브리프를 씌우는 비용이 요청보다 큽니다. 반대로 이슈로 추적할 가치가 있는 비자명 작업이면
브리프가 값을 합니다.

## 브리프 포맷 (4섹션 + 선택 예산)

```markdown
## 작업
<한 줄 사실 목표 — "무엇을"이 참·거짓으로 읽히게>

## 범위
- 포함: <건드릴 경로·시스템·함수 = sandbox>
- 제외: <건드리지 않을 것 — 인접 작업·계약·리팩터 유혹>

## 검증 방법
- <명령·파일·수치 — 예: `pytest tests/` exit==0>
- <예: report.md 에 합계 행 존재, 항목 수 == 12>

## 완료 정의
- <관측 가능한 상태 — 예: 위 테스트 GREEN, 결과 파일 생성됨>

## 예산 (선택)
- 질문 허용: <N회 — 에이전트가 되물을 수 있는 상한>
- 토큰/시간 상한: <예: 200k / 30분>
```

`## 검증 방법: ...` 같은 라벨 형태도 린터가 인식하지만, 헤딩 형태를 권장합니다.

## 절차

### 채점 모드

1. 초안을 임시 파일(예: `brief.md`)로 저장합니다.
2. 구조 게이트를 실행합니다:
   ```bash
   # macOS/Linux
   python3 scripts/check_task_brief.py <brief.md>
   # Windows
   py -3 scripts/check_task_brief.py <brief.md>
   ```
3. **구조 게이트(기계)** 는 C1~C5 를 강제합니다: 4섹션 존재·범위 경계(포함+제외)·검증
   재현성(명령·파일·수치 + 자기보고 어휘 0)·완료 상태 서술·모호어 0. W1(예산 권장)·
   W2(한 장 초과)는 경고입니다.
4. **결손 축마다 구체 보강 질문을 하나 이상** 만들어 사용자에게 되묻습니다. 게이트 detail 을
   질문으로 번역합니다:
   - C2 실패 → "제외(건드리면 안 되는 것)는 무엇입니까? 예: 인접 작업·다른 파일·리팩터."
   - C3 실패(자기보고) → "'확인했다' 대신 무엇을 **돌려서** 판정합니까? 명령/파일/수치로."
   - C4 실패 → "끝났을 때 파일/테스트/상태에서 **관측되는** 것은 무엇입니까?"
5. **의미 점검(에이전트)** — 구조 게이트가 못 잡는 것을 직접 채점합니다:
   - 검증이 정말 **재현 가능**한가(받는 쪽이 그 명령만으로 판정 가능한가).
   - 범위 제외가 **진짜 유혹**을 막는가("겸사겸사 리팩터"·인접 작업까지 번지는 경로).
   - 완료 정의가 **완결**인가(결과물·정리까지, 아니면 중간에서 그치는가).

### 인터뷰 모드

초안이 없으면 3요소를 순서대로 묻습니다. **한 번에 한 질문, 각 질문에 추천 기본값을 제시**합니다
— 사용자는 기본값을 수락하거나 고치기만 하면 됩니다. 모호한 답은 즉시 명령·파일·수치로 되묻습니다.

1. **작업** — 무엇을 시키려 합니까? (한 줄. "잘 되게"류는 "무엇이 관측되면 된 것인가"로 되묻기)
2. **범위 — 포함** — 어느 파일·경로·시스템을 건드려야 합니까?
3. **범위 — 제외** — 건드리면 안 되는 것은? (추천: 인접 작업·다른 계약·리팩터 유혹)
4. **검증** — 무엇을 **돌려서** 됐다고 판정합니까? (명령·파일·수치 — 자기보고 금지)
5. **완료** — 끝났을 때의 관측 가능한 상태는? (추천: 테스트 GREEN·결과 파일 생성·정리)
6. (선택) **예산** — 되물음 허용 횟수·토큰/시간 상한은?

답을 4섹션 포맷으로 조립한 뒤 **채점 모드 2~5단계로 검증**합니다.

## 마무리 — 그대로 던질 수 있는 한 장

구조 게이트 통과 + 의미 점검 통과 시, 브리프를 사용자에게 제시합니다:

```
아래 브리프를 그대로 붙여 작업을 던지세요:

<4섹션(+예산) 브리프 전문>
```

## 경계 — 언제 다른 흐름으로 넘기나

task-brief 는 **일상 요청 한 건**(SPEC 급은 아닌)을 위한 경량 계층입니다. 규모가 커지면
넘깁니다:

- 요청이 **자율 루프**(멈추지 않고 조건 충족까지 도는 목표)라면 → 목표 조건 규율 도구로.
  task-brief 의 검증·완료는 사람이 읽는 계약입니다.
- 요청이 **SPEC 급**(공개 계약 변경·다항목 요청·설계 변경, W2 경고 또는 브리프가 7항목 초과)
  이면 → SPEC 워크플로우로. 브리프 한 장으로 담기지 않습니다.
- **자명한 한 줄 요청**이면 → 발동하지 않습니다.

## 한계

- 구조 게이트는 **형식**만 강제합니다. "검증이 정말 재현 가능한가·범위 제외가 진짜 유혹을
  막는가"는 사람/에이전트의 의미 점검 몫입니다 — 스크립트 통과 ≠ 좋은 브리프.
- 자기보고·모호어 블랙리스트는 유한합니다. 새 어휘는 발견 시 `scripts/check_task_brief.py` 의
  `SELF_REPORT_TERMS`·`VAGUE_TERMS` 에 추가합니다.
- **soft 강제** — 이 스킬을 거치지 않고 요청을 바로 던지면 규율이 적용되지 않습니다. 값은
  "던지기 전 30초"를 습관으로 만드는 데 있습니다.
